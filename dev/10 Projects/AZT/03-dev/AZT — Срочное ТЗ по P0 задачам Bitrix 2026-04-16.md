# AZT — Срочное ТЗ по P0 задачам Bitrix

Дата: 16 апреля 2026  
Основание: [[AZT — Итоговый анализ данных Яндекс Вебмастер 2026-04-16]]  
Связанные документы: [[AZT — P0 задачи с обоснованием из Вебмастера 2026-04-16]], [[AZT — ТЗ для разработчика Bitrix 2026-04-13-codex]]

## Назначение документа

Документ описывает срочные работы по Bitrix и смежным настройкам аналитики.  
Для каждой задачи указаны проблема, состав работ, ожидаемый результат и критерии приёмки.

Данные Яндекс.Метрики на момент подготовки ТЗ не выгружались. Восстановление связки Метрики и Вебмастера вынесено в P0, чтобы после технических правок можно было проверить визиты, цели и заявки.

Цель работ:

- остановить генерацию битых URL;
- стабилизировать индексацию;
- привести региональные URL и canonical к единой модели;
- пересобрать sitemap после исправления источников ошибок;
- восстановить корректный мониторинг результата.

## 2. Задача P0.1 — восстановить связку Метрики и Вебмастера

Проблема:

| Данные | Значение |
|---|---|
| Диагностика | `NO_METRIKA_COUNTER_BINDING` |
| Severity | `POSSIBLE_PROBLEM` |
| State | `PRESENT` |
| Дата | `2026-04-13T08:00:45.483+03:00` |
| Нули в поисковых запросах | `2026-04-07` — `2026-04-16` |

Что сделать:

- проверить, установлен ли актуальный счётчик Метрики на сайте;
- привязать счётчик Метрики к сайту в Яндекс.Вебмастере;
- проверить права доступа между Метрикой, Вебмастером и Директом;
- проверить цели: звонок, форма, клик по телефону, мессенджер, квиз, заказ обратного звонка;
- зафиксировать ID счётчика и список активных целей.

Ожидаемый результат:

- Вебмастер видит связанный счётчик Метрики;
- диагностика `NO_METRIKA_COUNTER_BINDING` исчезает после обновления данных;
- дальнейшие SEO и рекламные правки можно оценивать по визитам, целям и заявкам.

Критерий приёмки:

- в Вебмастере нет активного `NO_METRIKA_COUNTER_BINDING`;
- счётчик Метрики установлен, активен и принимает тестовый визит;
- цели работают на тестовой заявке;
- ответственный передал ID счётчика и список целей.

Важно:

- если у разработчика нет доступа к Метрике и Вебмастеру, задачу выполняет агентство или владелец доступа;
- разработчик участвует только если счётчик отсутствует в коде сайта или установлен некорректно.

## 3. Задача P0.2 — исправить генерацию битых внутренних ссылок

Проблема:

Вебмастер фиксирует `428` битых внутренних ссылок. Основные паттерны:

| Паттерн | Количество |
|---|---:|
| `catalog/catalog` | `242` |
| `/prm/` | `93` |
| статьи без `/news/` | `4` |
| отсутствующий `og-default.jpg` | `34` |

Примеры неправильных URL:

```text
/prm/catalog/catalog/teplicy/arochnye/teplitsa-azt-lyuks/
/prm/catalog/catalog/teplicy/arochnye/teplitsa-azt-premium/
/catalog/catalog/teplicy/arochnye/teplitsa-azt-standart/
/kak-vybrat-teplitsu-iz-polikarbonata-podrobnoe-rukovodstvo-pered-pokupkoy/
```

Правильная логика:

```text
/catalog/teplicy/arochnye/teplitsa-azt-lyuks/
/catalog/teplicy/arochnye/teplitsa-azt-premium/
/catalog/teplicy/arochnye/teplitsa-azt-standart/
/news/kak-vybrat-teplitsu-iz-polikarbonata-podrobnoe-rukovodstvo-pered-pokupkoy/
```

Что сделать:

- найти компоненты и шаблоны Bitrix, которые генерируют ссылки на товары, связанные товары, хиты, похожие товары, категории и статьи;
- убрать удвоение сегмента `catalog/catalog`;
- убрать ошибочную подстановку `/prm/` в товарные URL;
- исправить ссылки на статьи: все статьи должны вести через `/news/...`;
- восстановить файл `https://azt-teplica.ru/local/templates/azt/images/og-default.jpg` или заменить все ссылки на существующий файл;
- проверить главную, категории, карточки товаров, региональные страницы, футер, breadcrumbs, sitemap-шаблоны.

Обоснование срочности:

- Bitrix формирует битую ссылочную массу: внутренние блоки сайта и sitemap передают Яндексу URL, которых не должно быть в рабочей структуре;
- при обходе таких URL Яндекс получает редиректы, 404-ответы, неканонические страницы и исключения из поиска;
- при добавлении новых товаров, статей и блоков ошибка будет воспроизводиться, если не исправить шаблоны;
- пересборка sitemap имеет смысл только после исправления генераторов URL.

Ожидаемый результат:

- сайт перестаёт генерировать битые внутренние ссылки;
- новые товары и статьи не создают новые ошибочные URL;
- робот Яндекса не расходует обход на несуществующие страницы;
- пользователи не попадают на 404 из внутренних блоков сайта.

Критерий приёмки:

- в HTML главной нет `catalog/catalog`;
- в HTML ключевых категорий нет `catalog/catalog`;
- в HTML карточек товаров нет `catalog/catalog`;
- ссылки на статьи на главной и в блоках статей ведут на `/news/...`;
- `og-default.jpg` отдаёт `200` или ссылки на него удалены;
- сканирование 20 ключевых страниц не показывает внутренних ссылок на `404`.

Команды для выборочной проверки:

```bash
curl -s https://azt-teplica.ru/ | rg 'catalog/catalog|/prm/catalog|href="/[^"]*kak-vybrat|og-default'
curl -I https://azt-teplica.ru/local/templates/azt/images/og-default.jpg
```

## 4. Задача P0.3 — проверить региональность и внедрить единую региональную URL-модель

Проблема:

Вебмастер фиксирует `22` события `NOT_CANONICAL`. Часть региональных URL канонизируется в корневые URL, поэтому Яндекс не воспринимает эти региональные страницы как самостоятельные.

Примеры:

| Региональный URL | Какой URL выбран каноническим |
|---|---|
| `/izh/` | `/` |
| `/izh/delivery/` | `/delivery/` |
| `/izh/portfolio/` | `/portfolio/` |
| `/izh/catalog/.../teplitsa-azt-premium/` | корневой товарный URL |
| `/ekb/actions/rassrochka/` | `/actions/rassrochka/` |

Подтверждённые региональные события из Вебмастера:

| Региональный паттерн | События в sample | Что это значит |
|---|---:|---|
| `/izh/` | `14` событий `NOT_CANONICAL` | Ижевские URL уже исключаются из поиска как неканонические |
| `/prm/` | `6` событий `LOW_QUALITY` | Пермские региональные URL попали в зону риска по качеству страниц |
| `/prm/` | `60` событий `REDIRECT_NOTSEARCHABLE` | Пермские региональные URL массово редиректят на корневые URL и не закрепляются как самостоятельные страницы |
| `/ekb/` | `1` событие `NOT_CANONICAL` | Аналогичный риск уже виден по Екатеринбургу |
| `/tmn/` | `0` таких событий в sample | В выгрузке нет подтверждения проблемы по Тюмени, но при той же шаблонной логике риск аналогичный |

Специфика Яндекса:

- регион сайта можно указать или изменить в Яндекс.Вебмастере;
- регион также может подтягиваться из Яндекс Бизнеса;
- после изменения региона Яндекс проверяет соответствие выбранного региона данным на сайте;
- контакты, адреса, телефоны, условия доставки и региональная информация на сайте обязательны для прохождения модерации;
- если компания работает в нескольких регионах, это нужно подтверждать адресами/телефонами на сайте и карточками в Яндекс Бизнесе.

Целевая модель:

- региональная модель нужна;
- список индексируемых региональных страниц утверждается после проверки фактической географии, контактов, доставки и Яндекс Бизнеса;
- региональность в Яндекс.Вебмастере и Яндекс Бизнесе должна соответствовать фактической географии бизнеса;
- для индексируемых региональных страниц canonical должен указывать на саму региональную страницу;
- title, description, H1, локальный текст, контакты, доставка и sitemap должны соответствовать региону;
- если региональная страница не предназначена для индексации, она не должна попадать в sitemap и внутреннюю перелинковку как SEO-страница.

Что сделать:

- проверить текущую региональность сайта в Яндекс.Вебмастере: Представление в поиске → Региональность;
- проверить карточку организации в Яндекс Бизнесе: адрес сайта, основной URL, адреса, телефоны, регионы работы;
- выбрать целевую региональную модель: один регион, регион «Россия», несколько подтверждённых представительств или региональные страницы;
- подготовить на сайте подтверждающие данные под выбранную модель: адреса, телефоны, доставку, региональные страницы, контакты;
- указать или скорректировать регион в Яндекс.Вебмастере после подготовки подтверждающих данных, если текущая настройка не соответствует фактической модели;
- зафиксировать список регионов, которые должны индексироваться;
- для индексируемых региональных URL сделать self-canonical;
- убрать canonical с региональной страницы на корневую, если страница должна продвигаться как региональная;
- синхронизировать title, description, H1 и хлебные крошки с региональной логикой;
- вывести на сайте контактную и доставочную информацию, подтверждающую выбранную региональность;
- проверить региональные версии главной, доставки, портфолио, акций, категорий и карточек;
- подготовить sitemap так, чтобы он не противоречил canonical;
- при наличии региональных аналогов добавить корректный `hreflang`, если это предусмотрено SEO-логикой.

Ожидаемый результат:

- по каждому URL Яндекс получает непротиворечивые сигналы;
- региональность в Вебмастере, Яндекс Бизнесе и на сайте не противоречит друг другу;
- региональные страницы не конфликтуют с корневыми;
- sitemap и canonical не противоречат друг другу;
- региональные страницы подготовлены к дальнейшему SEO/GEO-продвижению.

Критерий приёмки:

- в Яндекс.Вебмастере проверена страница Представление в поиске → Региональность;
- в Яндекс Бизнесе проверены сайт, адреса, телефоны и регионы работы;
- у индексируемой региональной страницы canonical указывает на неё саму;
- у неиндексируемой региональной страницы нет противоречия между canonical, sitemap и внутренними ссылками;
- `/izh/`, `/ekb/`, `/tmn/` и ключевые региональные категории проверены вручную;
- новые `NOT_CANONICAL` по целевым региональным страницам не появляются после переобхода.

Пример:

```html
<!-- было на /izh/delivery/ -->
<link rel="canonical" href="https://azt-teplica.ru/delivery/">

<!-- должно быть, если /izh/delivery/ продвигается как региональная страница -->
<link rel="canonical" href="https://azt-teplica.ru/izh/delivery/">
```

## 5. Задача P0.4 — пересобрать sitemap после исправления URL-шаблонов

Проблема:

Вебмастер принимает `sitemap.xml` без XML-ошибок, но другие отчёты показывают проблемы URL-структуры: `HTTP_ERROR`, broken internal links, `NOT_CANONICAL`, источники из sitemap XML.

Дополнительная живая проверка ранее показала:

- `57` URL в 5 основных sitemap;
- `56` уникальных URL;
- `14` URL отдают `404`;
- среди проблемных URL есть `404.php`, `thank.php`, неверные URL статей и кейсов.

Что сделать:

- пересобрать sitemap только после выполнения P0.2 и P0.3;
- убрать все URL, которые отдают `404`, `410`, `500`;
- убрать служебные URL:
  - `/404.php`;
  - `/thank.php`;
  - success/thanks-страницы форм;
  - тестовые и технические URL;
- заменить URL статей без `/news/` на реальные URL с `/news/...`;
- добавить живые индексируемые страницы, которые отсутствуют в sitemap:
  - `/contacts/`;
  - `/delivery/`;
  - `/portfolio/`;
  - `/reviews/`;
  - `/catalog/polikarbonat/`;
  - `/catalog/gryadki/`;
  - `/actions/`;
  - `/policy/`;
  - `/privacy/`;
- проверить все child-sitemap по HTTP-статусам.

Ожидаемый результат:

- sitemap содержит только индексируемые URL со статусом `200`;
- sitemap не содержит служебные страницы;
- sitemap не содержит URL, canonical которых указывает на другой URL;
- sitemap не содержит URL с ошибочной региональной логикой.

Критерий приёмки:

- все URL из sitemap отдают `200`;
- нет URL из sitemap со статусами `3xx`, `4xx`, `5xx`;
- нет `/404.php` и `/thank.php`;
- статьи в sitemap идут через `/news/...`;
- sitemap переотправлен на проверку ответственным за Вебмастер.

Команда для автоматической проверки должна вывести только `200`:

```bash
# пример логики проверки, точную команду адаптировать под структуру sitemap
curl -s https://azt-teplica.ru/sitemap.xml
```

## 6. Задача P0.5 — исправить 404, thank/success и служебные страницы

### 6.1. 404-шаблон

Проблема:

- 404-страницы отдают HTTP `404`, что корректно;
- при этом в HTML были обнаружены признаки обычной индексируемой страницы;
- старые meta-данные на 404-шаблоне не должны попадать в индекс или сниппеты.

Что сделать:

- сохранить HTTP-статус `404`;
- добавить `<meta name="robots" content="noindex, follow">`;
- убрать мусорные title, description, keywords;
- убрать canonical с 404-страницы;
- проверить, что 404-страница не попадает в sitemap.

Критерий приёмки:

```bash
curl -I https://azt-teplica.ru/never-existing-test-url/
curl -s https://azt-teplica.ru/never-existing-test-url/ | rg 'robots|canonical|keywords|description|title'
```

Ожидаемо:

- HTTP status `404`;
- есть `noindex`;
- нет canonical;
- нет старых нерелевантных meta.

### 6.2. thank/success страницы

Проблема:

Служебные страницы после отправки форм не должны индексироваться и не должны попадать в sitemap.

Что сделать:

- найти все thank/success URL;
- добавить `noindex`;
- исключить из sitemap;
- убрать внутренние ссылки на региональные или ошибочные URL со страниц благодарности.

Критерий приёмки:

- thank/success страницы содержат `noindex`;
- thank/success страницы отсутствуют в sitemap;
- внутренние ссылки на таких страницах ведут на существующие URL.

## 7. Задача P0.6 — проверить рекламные URL с `erid`, UTM и посадочные страницы

Проблема:

В Search Events есть URL с `?erid=...` и `bad_http_status = 404`. Это означает, что часть промаркированных или рекламных URL могла попадать на несуществующие страницы.

Подтверждённый пример из Вебмастера:

| URL | Event | Event date | Last access | Status |
|---|---|---|---|---|
| `https://azt-teplica.ru/catalog/teplicy/teplitsa-azt-lyuks/?erid=3MtFQRkqfB2huzBcHHttGDZbm2gFPT` | `REMOVED_FROM_SEARCH` | `2026-04-09` | `2026-03-29` | `HTTP_ERROR`, `bad_http_status = 404` |

Другие `erid`-URL в выгрузке:

| URL | Event | Статус |
|---|---|---|
| `/catalog/teplicy/pryamostennye/teplitsa-azt-pavilon/?erid=3MtFQRkqfB2huzBcHHttGDZbm2faGn` | `REMOVED_FROM_SEARCH` | `NOT_CANONICAL` |
| `/parnik-domik?erid=3MtFQRkqfB2huzBcHHttGDZbm2faM5` | `REMOVED_FROM_SEARCH` | `REDIRECT_NOTSEARCHABLE` |
| `/pavilyon?erid=3MtFQRkqfB2huzBcHHttGDZbm2faGn` | `REMOVED_FROM_SEARCH` | `NOT_CANONICAL` |

Что сделать:

- выгрузить из Директа активные объявления и посадочные URL;
- проверить URL с `erid`, UTM и другими параметрами;
- убедиться, что каждый рекламный URL после редиректов открывает целевую страницу со статусом `200`;
- проверить, что параметры не меняют canonical некорректно;
- проверить, что `Clean-param` и canonical не конфликтуют;
- исправить объявления или редиректы, если URL ведёт на `404`.

Ожидаемый результат:

- платный трафик не ведётся на 404-страницы;
- рекламные параметры не создают индексируемые дубли;
- аналитика по рекламе и SEO не искажается ошибочными посадочными URL.

Критерий приёмки:

- список активных рекламных URL проверен;
- все активные посадочные URL отдают `200`;
- URL с параметрами имеют корректный canonical на чистую посадочную страницу;
- нет активных объявлений, ведущих на 404.

## 10. Финальный критерий выполнения P0

P0 считается выполненным, когда:

- Вебмастер не показывает `NO_METRIKA_COUNTER_BINDING`;
- сайт не генерирует `catalog/catalog`;
- сайт не генерирует товарные URL через ошибочный `/prm/catalog/catalog/...`;
- ссылки на статьи ведут через `/news/...`;
- `og-default.jpg` либо отдаёт `200`, либо больше не используется;
- региональность в Яндекс.Вебмастере/Яндекс Бизнесе проверена и согласована с сайтом;
- региональные страницы имеют согласованную canonical-модель;
- sitemap содержит только URL со статусом `200`;
- sitemap не содержит служебных URL;
- 404-страница отдаёт `404`, содержит `noindex` и не содержит canonical;
- thank/success страницы закрыты от индексации и отсутствуют в sitemap;
- активные рекламные URL не ведут на `404`;
- после переобхода Вебмастера счётчики broken links, `HTTP_ERROR` и `NOT_CANONICAL` начинают снижаться.
