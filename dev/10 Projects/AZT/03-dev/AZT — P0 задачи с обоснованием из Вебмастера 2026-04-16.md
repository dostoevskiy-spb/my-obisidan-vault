## 1. Привязать Яндекс.Метрику к Вебмастеру

**Проблема.** Счётчик Метрики (87177110) установлен на сайте, но не привязан к аккаунту Вебмастера. Вебмастер сигнализирует `NO_METRIKA_COUNTER_BINDING` (severity: POSSIBLE_PROBLEM, зарегистрировано 2026-04-13).

Счётчик подтверждённо отсутствует в кабинете Вебмастера.

**Последствия.**
- Вебмастер не использует данные Метрики для диагностики и рекомендаций.
- Данные о поведении пользователей не учитываются в ранжировании.
- С высокой вероятностью связано с тем, что данные по кликам/показам в Вебмастере = 0 с 7 апреля (см. задачу мониторинга в конце).
- Без привязки невозможно отслеживать результат остальных P0-фиксов через Вебмастер.

**Что сделать.** В Яндекс Вебмастере: Настройки → Привязка Яндекс.Метрики → привязать счётчик 87177110. Нужен доступ к обоим сервисам от одного аккаунта (или делегированный).

**Критерий.** `NO_METRIKA_COUNTER_BINDING` переходит из PRESENT в ABSENT.

## 2. Canonical главной: убрать двойной слеш

**Проблема.** Canonical на главной: `https://azt-teplica.ru//` (двойной слеш). Должно быть: `https://azt-teplica.ru/`.

**Последствия.**
- Двойной слеш — технически другой URL. Яндекс может считать `//` и `/` разными страницами.
- Региональные главные (`/ekb/`, `/tmn/`, `/izh/`) канонизируются в этот сломанный URL. Вся цепочка канонизации невалидна.

**Данные.** В Search Events: `/izh/` исключена как `NOT_CANONICAL` с target_url `https://azt-teplica.ru/`. Яндекс сам исправил двойной слеш при обработке, но вынужден "догадываться", а не получать чистый сигнал.

**Что сделать.** В шаблоне Битрикс: найти место генерации canonical для главной, убрать двойной слеш.

**Критерий.** `curl -s https://azt-teplica.ru/ | grep canonical` → `https://azt-teplica.ru/` (один слеш).

## 3. Закрыть `thank.php` от индексации

**Проблема.** Служебная страница подтверждения отправки формы (`thank.php`) отдаёт 200, не закрыта от индексации.

**Данные.**
- В списке битых внутренних ссылок: `thank.php` → `/prm/` (битая, обнаружена 21.03).
- В Search Events: `/izh/thank.php` с title "Title" появлялась в поиске.

**Что сделать.**
1. Добавить `<meta name="robots" content="noindex, nofollow">` в шаблон `thank.php`.
2. Ссылку на `/prm/` заменить на `/` или убрать.

**Критерий.** `thank.php` содержит meta noindex. В Вебмастере через 2-4 недели: `/thank.php` и `/izh/thank.php` исключены из поиска.

## 4. Исправить генерацию ссылок `/prm/catalog/catalog/...`

**Проблема.** Шаблоны Битрикс генерируют внутренние ссылки с удвоенным сегментом:
```
/prm/catalog/catalog/teplicy/arochnye/teplitsa-azt-lyuks/
```
вместо:
```
/catalog/teplicy/arochnye/teplitsa-azt-lyuks/
```

Эти ссылки ведут на 301 → 404 или сразу 404. Каждая карточка товара генерирует такие ссылки на все "связанные товары".

**Данные.**
- 428 битых внутренних ссылок на 15 апреля. Больше половины — паттерн `/prm/catalog/catalog/...`.
- Рост: 72 (14.03) → 112 (25.03) → 213 (03.04). Удвоение каждые 9-10 дней. Каждый новый товар добавляет новые битые ссылки.
- HTTP_4XX при обходе: до 142 ошибок в день (15.03), стабильно 4-80 в день в апреле.
- Яндекс исключает `/prm/`-версии как LOW_QUALITY: `/prm/catalog/komplektuyushchie/komplekt-6-7-peremychka/` (09.04), `/prm/catalog/teplicy/arochnye/teplitsa-azt-20-40/` (14.04).
- Страницы в поиске падают: 165 (05.04) → 138 (14.04). Минус 27 за 11 дней.

Конкретные примеры из API:
- Главная → `/prm/catalog/catalog/teplicy/arochnye/teplitsa-azt-lyuks/`
- Главная → `/prm/catalog/catalog/teplicy/arochnye/teplitsa-azt-premium/`
- Карточка АЗТ Люкс → `/catalog/catalog/teplicy/arochnye/teplitsa-azt-standart/` (дублируется 3 раза)
- Карточка АЗТ 20×40 → `/prm/catalog/catalog/teplicy/arochnye/teplitsa-azt-fermerskaya-lyuks/` (дублируется 3 раза)

**Почему первоочерёдно.** Единственная проблема из P0, которая растёт сама. Sitemap статичен — 14 битых URL одни и те же. А битые внутренние ссылки множатся при каждом обходе и при добавлении каждого нового товара.

**Что сделать.** Найти в шаблонах компонент, генерирующий ссылки на "связанные товары" / "похожие" / "хиты". Исправить логику построения URL. Проверить: главная, блоки хитов, категории, карточки товаров, региональные страницы, футер, breadcrumbs.

**Критерий.** `grep -r 'catalog/catalog' *.html` по шаблонам = 0 результатов. В Вебмастере через 2-4 недели: счётчик битых ссылок начинает сокращаться.

## 5. Исправить ссылки на статьи блога (без `/news/`)

**Проблема.** На главной и в шаблонах ссылки на статьи ведут по корневым URL без `/news/`:
- `/kak-vybrat-teplitsu-iz-polikarbonata-podrobnoe-rukovodstvo-pered-pokupkoy/` → 404
- `/usilennye-teplitsy-iz-polikarbonata-reshenie-dlya-regionov-so-snegovoy-nagruzkoy/` → 404
- `/kak-uvelichit-urozhaynost-v-teplitse-prakticheskie-metody-dlya-maksimalnogo-rezultata/` → 404
- `/teplitsy-dlya-dachi-kakuyu-model-vybrat-dlya-komfortnogo-vyrashchivaniya/` → 404

Реальные статьи живут по `/news/...` и отдают 200.

**Данные.** В списке 428 битых внутренних ссылок 4 URL статей идут с главной (`source_url: https://azt-teplica.ru/`), все обнаружены 22.03. Те же 4 URL перечислены в `sitemap-iblock-9.xml` — Яндекс видит их и в sitemap, и на главной, и оба раза получает 404.

**Почему сейчас.** Агентство пишет 15 новых статей. Если шаблон не починить до публикации, новые статьи попадут в тот же паттерн: ссылки без `/news/`, статьи не получат ни внутреннего веса с главной, ни корректной записи в sitemap.

**Что сделать.** Найти в шаблоне (главная, блок "Статьи"/"Новости") компонент генерации ссылок на статьи. Заменить URL на `/news/...`.

**Критерий.** Все ссылки на статьи в HTML главной ведут на `/news/...`. В `sitemap-iblock-9.xml` URL статей с префиксом `/news/`.

## 6. Пересобрать sitemap

**Проблема.** Sitemap содержит 14 битых URL из 56 уникальных. 9 живых страниц из навигации в sitemap отсутствуют. Страница `404.php` включена в sitemap.

**Данные.**
- Sitemap: 1 индексный, 20 child sitemaps. Вебмастер показывает 0 ошибок (проверяет XML-валидность, не доступность URL).
- В Search Events за апрель 16+ URL исключены из поиска как HTTP_ERROR:
  - `/catalog/teplicy/teplitsa-azt-20-30/` — 404, исключена 08.04
  - `/catalog/teplicy/teplitsa-azt-premium/` — 404, исключена 08.04
  - `/catalog/teplicy/teplitsa-azt-kottedzh/` — 404, исключена 08.04
  - `/catalog/teplicy/teplitsa-azt-fermerskaya-next/` — 404, исключена 12.04
  - `/catalog/teplicy/teplitsa-azt-kapelyusha-lyuks/` — 404, исключена 12.04
  - `/catalog/teplicy/azt-lyuminiy-pod-steklo-2-2-4-metra/` — 404, исключена 12.04
  - и ещё ~10 URL
- 14 битых URL × 4 региона = 56 битых записей в общей sitemap-структуре.

**Что сделать.**
1. Убрать все 404 из sitemap (14 URL из sitemap-files, sitemap-iblock-9, sitemap-iblock-12, sitemap-iblock-20).
2. Убрать `/404.php` из sitemap-files.
3. Заменить 4 article URL в sitemap-iblock-9 на реальные `/news/...`.
4. Добавить 9 живых страниц: `/contacts/`, `/delivery/`, `/portfolio/`, `/reviews/`, `/catalog/polikarbonat/`, `/catalog/gryadki/`, `/actions/`, `/policy/`, `/privacy/`.

Snapshot текущих sitemaps для сверки: [[_baseline/sitemaps-2026-04-15/README]]

**Критерий.** Все URL из sitemap отдают 200. Ни один URL из sitemap не ведёт на 3xx, 4xx, 5xx. Sitemap перезагружен в Вебмастере.

## 7. Исправить 404-шаблон

**Проблема.** Страницы с ошибкой 404:
- Отдают HTTP 404 (корректно)
- Содержат `meta robots="index, follow"` (противоречит HTTP 404)
- Содержат мусорные title, description, keywords от чужого сайта/шаблона
- Содержат невалидный canonical

**Данные.** В Search Events за апрель 16+ страниц исключены как HTTP_ERROR. Диагностика `SOFT_404` сейчас ABSENT (последнее срабатывание 23.02), но мусорные meta на 404-шаблоне могут спровоцировать возврат проблемы.

**Что сделать.** В шаблоне 404 Битрикса:
1. Добавить `<meta name="robots" content="noindex, follow">`.
2. Убрать мусорные title, description, keywords.
3. Убрать canonical.

**Критерий.** Любая несуществующая страница: HTTP 404, в HTML есть `noindex`, нет мусорных meta, нет canonical.

## Мониторинг: нули в данных Вебмастера с 7 апреля

**Ситуация.** С 7 апреля 2026 в Вебмастере нули по кликам и показам. Данные за 6 апреля подгрузились с задержкой (вчера через API = 0, сегодня = 66 кликов / 370 показов). С 7 по 16 апреля — 10 дней нулей.

При этом обход сайта идёт нормально: Searchable pages обновляются (138 на 14.04), Search Events обновляются, Indexing history обновляется (113 HTTP_2XX на 14.04). Нули только в данных по поисковым запросам.

**Текущая оценка.** Скорее всего, причина — разрыв связки Метрика↔Вебмастер (счётчик подтверждённо отвалился, диагностическая ошибка `NO_METRIKA_COUNTER_BINDING` зарегистрирована 13.04). Подтверждается тем, что данные за 6 апреля подгрузились задним числом — Яндекс продолжает собирать статистику, но отдаёт с задержкой. Ограничений и санкций в кабинете Вебмастера нет. Нормальные сроки обновления данных Вебмастера — до 2 недель.

**Действия.**
1. Выполнить задачу 1 (привязка Метрики) — это может восстановить поток данных.
2. Подождать до 22-25 апреля. Если данные вернутся — лаг, всё в порядке.
3. Если данные не вернутся после привязки + 2 недели ожидания — обращаться в техподдержку Яндекса.
