# AZT — Детализированный план исправления ошибок для разработчика

Дата: 2026-04-14  
Основа: живая перепроверка `azt-teplica.ru` + итоговый отчёт для собственника

## Назначение документа

Это прикладной план для разработчика и технического подрядчика.  
Цель документа:

- превратить выводы аудита в рабочий список внедрения
- зафиксировать порядок исправлений
- показать, что именно сейчас сломано
- описать, каким должен быть результат
- дать понятный критерий приёмки по каждому блоку

Документ сознательно сфокусирован на том, что подтверждено по живому сайту.

---

## Общая логика внедрения

Порядок работ должен быть таким:

1. сначала починить индексационный фундамент
2. затем привести к одной логике региональность и мета-слой
3. потом добавлять schema и улучшать шаблоны
4. после этого заниматься скоростью, trust-страницами и контентным усилением

Если менять этот порядок, можно потратить время на улучшение страниц, которые всё равно будут описаны сайтом для Яндекса неправильно.

### Задача P0.1 — Пересобрать sitemap

#### Что подтверждено сейчас

В текущих sitemap есть битые и служебные URL:

- `https://azt-teplica.ru/404.php`
- `https://azt-teplica.ru/thank.php`
- `https://azt-teplica.ru/actions/rassrochka/`
- `https://azt-teplica.ru/actions/khranenie/`
- `https://azt-teplica.ru/actions/dostavka/`
- `https://azt-teplica.ru/kak-vybrat-teplitsu-iz-polikarbonata-podrobnoe-rukovodstvo-pered-pokupkoy/`
- `https://azt-teplica.ru/promyshlennaya-teplitsa/`

Отдельно подтверждено:

- в основных sitemap сейчас `57` URL
- `56` из них уникальны
- `14` URL отдают `404`

Примеры:

- `sitemap-files.xml` содержит `/404.php` и `/thank.php`
- `sitemap-iblock-9.xml` содержит статьи без `/news/`
- `sitemap-iblock-20.xml` содержит кейсы без `/portfolio/`

#### Что сделать

- проверить логику генерации всех sitemap в Bitrix
- исключить из sitemap:
  - страницы со статусом `404`
  - страницы со статусом `410`
  - служебные страницы
  - thank/success URL
- заменить news URL без `/news/` на реальные URL с `/news/`
- заменить кейсовые URL без `/portfolio/` на реальные URL с `/portfolio/`
- добавить в sitemap живые страницы, которых сейчас нет:
  - `/contacts/`
  - `/delivery/`
  - `/portfolio/`
  - `/reviews/`
  - `/catalog/polikarbonat/`
  - `/catalog/gryadki/`
  - `/policy/`
  - `/privacy/`

#### Как должен выглядеть результат

- в sitemap остаются только реальные индексируемые `200`-страницы
- `404.php` и `thank.php` не присутствуют
- статьи публикуются в sitemap по URL `/news/...`
- кейсы публикуются в sitemap по URL `/portfolio/...`
- sitemap отражает фактическую структуру сайта, а не старые или ошибочные пути

#### Почему это важно

Именно sitemap сообщает поисковику, какие страницы сайт считает актуальными.  
Если sitemap содержит ошибки, служебные URL и неверные пути, Яндекс получает изначально грязную структуру сайта.

#### Критерий приёмки

- каждый URL из sitemap отдаёт `200`
- в sitemap нет `3xx`, `4xx`, `5xx`
- в sitemap нет `404.php`
- в sitemap нет `thank.php`
- news URL соответствуют реальным статьям
- portfolio URL соответствуют реальным кейсам

### Задача P0.2 — Исправить 404-шаблон

#### Что подтверждено сейчас

На `https://azt-teplica.ru/404.php`:

- HTTP-статус `404` корректный
- но есть `meta robots="index, follow"`
- есть canonical на саму 404-страницу
- есть чужой `description` про одежду
- есть чужие `keywords` про одежду и тапочки

Текущий meta-фрагмент:

```html
<meta name="robots" content="index, follow" />
<meta name="description" content="Мы предлагаем широкий ассортимент качественной одежды по адекватным ценам." />
<meta name="keywords" content="женская одежда, мужская одежда, спортивная одежда, тапочки" />
<link rel="canonical" href="https://azt-teplica.ru/404.php/" />
```

#### Что сделать

- оставить HTTP-статус `404`
- выставить `meta robots="noindex, follow"`
- убрать `description` и `keywords`, не относящиеся к сайту
- убрать canonical с 404-шаблона
- привести `title` к нормальному техническому виду

#### Как должен выглядеть результат

- 404-страница не индексируется
- в шаблоне отсутствует чужой контент
- canonical отсутствует
- 404 остаётся полезной для пользователя, но не воспринимается как контентная страница

#### Почему это важно

Сейчас сайт сам просит поисковик индексировать ошибку, причём с нерелевантной тематикой. Это технический мусор и плохой сигнал качества.

#### Критерий приёмки

- любой несуществующий URL отдаёт `404`
- в HTML есть `noindex, follow`
- нет `description` про одежду
- нет `keywords` про одежду
- нет canonical на 404

## Блок P1 — сразу после P0

### Задача P1.1 — Привести региональные страницы к одной логике

#### Что подтверждено сейчас

На региональных главных:

- `/ekb/`
- `/tmn/`
- `/izh/`

видно следующее:

- H1 локальный
- title пермский
- canonical ведёт на `https://azt-teplica.ru//`
- `hreflang` не найден

Пример с `/ekb/`:

```html
<title>Теплицы в Перми — купить от производителя...</title>
<link rel="canonical" href="https://azt-teplica.ru//" />
<h1>Производим и устанавливаем теплицы под ключ в Екатеринбурге</h1>
```

#### Что сделать

Сначала согласовать с SEO и собственником стратегию:

- либо один основной индексируемый регион
- либо полноценная мультирегиональность

Если выбран один основной регион:

- убрать конфликт между региональными копиями и основной версией
- настроить последовательную канонизацию
- не держать псевдорегиональные SEO-посадочные как полноценные документы

Если выбрана реальная мультирегиональность:

- сделать уникальные title/description/H1
- поставить self-canonical
- добавить локальные признаки
- затем добавить `hreflang`

#### Как должен выглядеть результат

- у каждой региональной страницы одна понятная логика
- title, H1 и canonical не конфликтуют друг с другом
- поисковику не отправляется противоречивый сигнал

#### Почему это важно

Сейчас страница выглядит как региональная только для пользователя, но не для поисковой логики.

#### Критерий приёмки

- `/ekb/`, `/tmn/`, `/izh/` не имеют пермского title
- canonical соответствует выбранной стратегии
- нет конфликтов между индексируемостью и канонизацией

### Задача P1.2 — Добавить `hreflang`, только если реально выбрана мультирегиональность

#### Что подтверждено сейчас

`hreflang` на региональных страницах не найден.

#### Что сделать

- добавить перекрёстные `hreflang` между версиями только если принято решение держать индексируемые региональные страницы
- проверить соответствие canonical и `hreflang`

#### Как должен выглядеть результат

- alternate-связи между версиями существуют
- они не конфликтуют с canonical

#### Почему это важно

`hreflang` не решает проблему вместо нормальной региональной архитектуры. Он нужен только после выравнивания логики регионов.

#### Критерий приёмки

- `hreflang` присутствует на региональных страницах, если выбран сценарий мультирегиональности
- `hreflang` отсутствует, если выбран сценарий одной канонической версии

#### Что подтверждено сейчас

#### Что сделать

- почистить шаблоны title/description
- убрать дубли формата `в Перми в Перми`
- убрать HTML entities из вывода
- проверить:
  - главную
  - категории
  - карточки
  - контакты
  - доставку
  - статьи

#### Как должен выглядеть результат

- title и description читаемы
- нет артефактов
- нет лишнего повторения региона

#### Почему это важно

Это влияет на качество сниппета и на восприятие сайта как аккуратно настроенного проекта.

#### Критерий приёмки

- на ключевых шаблонах title/description читаемы
- нет `&quot;`, `&amp;quot;` и подобных сущностей
