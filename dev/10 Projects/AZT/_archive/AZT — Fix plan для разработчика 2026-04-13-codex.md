# AZT — Fix plan для разработчика

Дата: 2026-04-13  
Основа: аудит `azt-teplica.ru` от 2026-04-12

## Цель документа

Это не общий аудит, а прикладной список исправлений для разработчика.  
Приоритет: сначала убрать всё, что ломает индексацию и краулинг, потом привести в порядок региональность, мета-слой, schema и производительность.

---

## 1. Критический блок: исправить в первую очередь

### 1.1 Починить sitemap

Нужно:

- убрать из sitemap все URL со статусом `404`
- убрать из sitemap `404.php`
- убрать из sitemap `thank.php`, если страница не должна индексироваться
- заменить article URL без `/news/` на реальные URL вида `/news/...`
- добавить в sitemap живые страницы, которых там сейчас нет:
  - `/contacts/`
  - `/delivery/`
  - `/portfolio/`
  - `/reviews/`
  - `/catalog/polikarbonat/`
  - `/catalog/gryadki/`
  - `/actions/`
  - `/policy/`
  - `/privacy/`

Проверка после исправления:

- каждый URL из sitemap должен отдавать `200`
- ни один URL из sitemap не должен вести на `3xx`, `4xx`, `5xx`
- sitemap должен соответствовать реальной структуре сайта

### 1.2 Исправить 404-шаблон

Нужно:

- для всех `404` оставить HTTP-статус `404`
- выставить `meta robots="noindex, follow"` минимум
- убрать мусорные `title`, `description`, `keywords`, не относящиеся к сайту
- убрать canonical на несуществующий URL, если сейчас он дублирует ошибочную страницу

Ожидаемый результат:

- поисковики перестанут воспринимать ошибки как валидные документы
- снизится риск индексации мусорных страниц

### 1.3 Закрыть служебные страницы от индексации

Проверить и исправить:

- `/thank.php`
- другие thank/submit/success страницы, если есть
- технические попап-страницы, если они доступны прямым URL

Нужно:

- или `noindex, nofollow`
- или исключение из sitemap и корректное закрытие на уровне шаблона

### 1.4 Исправить внутренние ссылки

Найденные проблемы:

- на главной есть ссылки на статьи без `/news/`
- встречаются ссылки вида `/prm/catalog/catalog/...`

Нужно:

- найти источник генерации ссылок в шаблонах Bitrix
- заменить все неправильные article URL на `/news/...`
- убрать генерацию дублированного сегмента `/catalog/catalog/`
- перепроверить ссылки в:
  - главной
  - новостном блоке
  - региональных версиях
  - хлебных крошках
  - карточках товара

## 2. Региональность: привести к одной логике

Сейчас реализована промежуточная схема: региональные URL есть, но SEO-сигналы частично канонизированы в Пермь.

Нужно сначала принять решение:

### Вариант A. Полноценные региональные посадочные

Тогда нужно:

- уникализировать title/description/H1 по регионам
- сделать корректные canonical на самих себя
- добавить `hreflang`
- добавить локальные контакты, адреса, склады, условия доставки
- добавить `LocalBusiness`/`Organization` schema по каждому региону

### Вариант B. Одна каноническая версия сайта

Тогда нужно:

- не индексировать псевдорегиональные копии как отдельные страницы
- настроить канонизацию и/или redirects последовательно
- убрать конфликт между региональными sitemap и общей канонической логикой

Рекомендация:

- если реально планируете органический трафик по Екатеринбургу, Тюмени, Ижевску, выбирайте `Вариант A`
- если регионы нужны только для UX-подстановки контактов, выбирайте `Вариант B`

### Технические задачи по региональности

- исправить title на `/ekb/`, `/tmn/`, `/izh/`
- исправить canonical на региональных главных
- добавить `hreflang`
- синхронизировать sitemap с принятой стратегией

## 3. Мета-слой и сниппеты

### 3.1 Canonical

Нужно:

- исправить canonical главной: убрать `//`
- проверить canonical на всех шаблонах:
  - главная
  - категории
  - карточки
  - статьи
  - региональные страницы
  - служебные страницы

Правило:

- canonical должен указывать только на существующий целевой URL
- canonical не должен указывать на битую страницу

### 3.2 Title / Description

Нужно:

- убрать избыточные шаблонные хвосты
- сократить длину title
- устранить дубли вида `в Перми в Перми`
- устранить артефакты размерности и форматирования
- не использовать одинаковые конструкции на всех страницах

Приоритетно пересобрать шаблоны для:

- главной
- категорий
- карточек
- контактов
- доставки
- отзывов
- портфолио
- статей

### 3.3 Open Graph

Нужно исправить как минимум:

- `/` — сейчас `og:title = Партнеры`
- `/about/` — сейчас `og:title = О магазине`
- `/delivery/` — сейчас `og:title = Title`
- статьи — использовать title статьи, а не общий title раздела

Также нужно:

- сделать нормальный `og:image` под шаблоны:
  - главная
  - категория
  - карточка товара
  - статья

### 3.4 Убрать технический мусор

Нужно:

- убрать `meta keywords="."`
- проверить дубли мета-тегов в шаблоне
- исправить HTML entities в мета-данных (`&quot;`, `&amp;quot;`)

## 4. Schema.org

### 4.1 Обязательно добавить

- `Organization`
- `LocalBusiness` или релевантный subtype
- `BreadcrumbList`

### 4.2 Для карточек товара усилить `Product`

Добавить:

- `description`
- `image`
- `brand`
- `sku` или внутренний артикул, если есть
- `offers.price`
- `priceCurrency`
- `availability`
- `url`
- `aggregateRating`, если есть реальные отзывы

Проверить:

- региональные данные в `eligibleRegion`
- корректность падежей и сущностей

### 4.3 Для статей добавить `Article` / `BlogPosting`

Добавить:

- `headline`
- `description`
- `author`
- `datePublished`
- `dateModified`
- `image`
- `mainEntityOfPage`
- `publisher`

### 4.4 Для FAQ

Если на карточках или в статьях есть ответы на типовые вопросы:

- добавить `FAQPage`

## 5. Контентные шаблоны: что нужно доработать на уровне кода и шаблонов

### 5.1 Категории

Нужно предусмотреть в шаблонах:

- нормальный SEO-текст без переоптимизации
- блок сравнения моделей
- блок FAQ
- блок преимуществ с фактами
- блок внутренней перелинковки на связанные товары и статьи

### 5.2 Карточки товаров

Добавить или усилить:

- блок теххарактеристик в структурированном формате
- блок "для кого подходит"
- блок "что входит в комплект"
- блок "что докупить сразу"
- блок FAQ
- блок доверия: гарантия, монтаж, доставка, хранение
- блоки отзывов и кейсов по конкретной модели

### 5.3 Статьи

На уровне шаблона статьи нужен вывод:

- автора / редактора
- даты публикации
- даты обновления
- времени чтения
- связанных товаров
- связанных статей

## 6. Производительность и техническая гигиена

### 6.1 Кэширование

Нужно разобраться, почему публичные страницы отдают:

- `cache-control: no-store, no-cache, must-revalidate`

Для продакшена это неверно почти наверняка.

Проверить:

- настройки Bitrix composite/cache
- прокси/серверный конфиг
- отключающие кэш модули и компоненты

Цель:

- статический HTML и ассеты должны нормально кэшироваться

### 6.2 Изображения

Нужно:

- включить `loading="lazy"` хотя бы на не-first-screen изображения
- проверить размеры и форматы
- использовать адаптивные источники, если возможно

### 6.3 Шаблонный JS/CSS шум

Нужно:

- сократить глобально подключаемые скрипты
- не тянуть лишние модули Bitrix на страницах, где они не используются
- проверить, что нет избыточных inline-блоков, дубликатов скриптов и CSS

## 7. GEO / AI readiness: технические задачи

Нужно:

- добавить `Organization` / `LocalBusiness`
- добавить `Article` schema
- добавить FAQ с разметкой
- сделать сущностно понятные блоки:
  - модель
  - размер
  - каркас
  - покрытие
  - гарантия
  - доставка
  - монтаж
  - регион
- при желании добавить `llms.txt`, если команда решит поддерживать отдельный AI-readable слой

Важно:

`llms.txt` не заменяет нормальную структуру сайта.  
Сначала нужно сделать чистую архитектуру URL, schema и контентных шаблонов.

## 9. Definition of done

Можно считать задачу закрытой, когда выполнены все условия:

- в sitemap нет `404`
- все статьи в sitemap ведут на реальные `/news/...`
- `404` не индексируются
- `thank.php` не индексируется
- canonical корректны
- на региональных страницах понятная и непротиворечивая логика
- OG корректен на ключевых шаблонах
- schema есть на компании, товарах, статьях
- внутренняя перелинковка не ведёт на мусорные URL
- публичные страницы не отключают кэш без причины

## 10. Практический вывод

Если делать только одну вещь первой, надо чинить `sitemap + 404 + внутренние ссылки`.  
Это самая дешёвая по трудозатратам и самая выгодная по эффекту группа исправлений.
