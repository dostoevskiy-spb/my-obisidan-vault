# AZT — Уроки из реального опыта выгрузок

Документ-дополнение к [[Регламент выгрузки и качества данных]]. Фиксирует конкретные случаи, которые мы встретили при работе с Яндекс Вебмастером, Яндекс Метрикой и счётчиками Яндекс Бизнеса. Каждый случай — симптом, причина, действие.

Регламент выгрузки Codex описывает правильный порядок работы. Этот документ — копилка исключений и специфических поведений API, на которые уже наступили.

## 1. Данные подгружаются задним числом

**Симптом.** Запросил через API `ywm_get_search_queries` за 6 апреля вчера — получил 0 кликов, 0 показов. Запросил сегодня за тот же день — 66 кликов, 370 показов.

**Причина.** Яндекс Вебмастер и Метрика обновляют данные с задержкой до 2 недель. Одни и те же даты могут отдавать разные значения на разных запусках.

**Правило работы.**
- Перед тем как делать вывод "данных нет" — перезапросить через день-два.
- Не фиксировать baseline по свежим датам, которые ещё могут измениться. Хвост в 3-7 дней считать "предварительным".
- В baseline-файлах указывать дату и время snapshot, чтобы понимать, на какой момент данные были актуальны.

**Пример.** [[AZT — Baseline Яндекс Вебмастер 2026-04-15]] — там указана дата 15.04. Цифры на 16.04 уже другие из-за подгрузки задним числом.

## 2. Разные endpoint-ы одного источника дают разные значения

**Симптом.** По одному и тому же периоду и счётчику:
- `search-queries/all/history`: 720 кликов, 4232 показа
- `query_analytics/list`: 500 кликов, 2976 показов

Разница ≈30% для одного и того же источника.

**Причина.** Разные endpoint-ы могут использовать разные агрегации, разные фильтры ботов, разный порядок обработки. Это не ошибка — это разная семантика данных.

**Правило работы.**
- Фиксировать, какой endpoint использовался для вывода.
- Сравнивать только одинаковые endpoint-ы между снимками.
- Для сегментации (бренд/коммерция, группы запросов) использовать detailed-выгрузку (`query_analytics/list`). Для трендовых метрик — history.
- Не складывать цифры из разных endpoint-ов в один KPI.

## 3. Смешение счётчика сайта и счётчиков карточек Яндекс Бизнеса

**Симптом.** В аккаунте Метрики видно 5 счётчиков AZT. Один — сайт (87177110). Остальные — автоматические счётчики карточек Яндекс Бизнеса (96563160 Пермь, 106684083 Екатеринбург, 106282002 Тюмень).

**Причина.** Счётчики карточек создаются Яндексом при регистрации организации на Картах. Они привязаны к `yandex.ru/maps`, не к сайту. Название и структура URL в них — это поверхности Яндекса, а не страницы сайта.

**Правило работы.**
- Перед анализом всегда проверять `site`/`site2` в свойствах счётчика.
- Признаки счётчика Яндекс Бизнеса:
  - `source: "sprav"`
  - `site: "yandex.ru/maps"`
  - `owner_login` вида `yndx-metrika-counters-NNN`
  - `permission: "view"` (нет прав редактирования)
- Для таких счётчиков не применять обычные поведенческие метрики (bounce, глубина, время на сайте) — они не про поведение на сайте. Использовать `vacuum`-срезы.
- Сайтовый счётчик (87177110) и счётчики карточек нужно связывать, но не складывать.

## 4. `query_analytics` отдаёт очень большие ответы

**Симптом.** Вызов `query_analytics` без limit или с limit=500 вернул файл 3.6 MB (~3.6 млн символов). Ответ превышает лимит токенов и сохраняется MCP-сервером в отдельный файл.

**Причина.** По каждому запросу отдаётся массив статистики по дням (POSITION, IMPRESSIONS, CTR, CLICKS, DEMAND). 1387 запросов × 30 дней × 5 метрик = десятки тысяч строк данных.

**Правило работы.**
- Сразу ограничивать `limit` (50-100 для первичного обзора).
- Для полной выборки сохранять ответ в файл `_data/` и парсить через `jq` чанками.
- Для каждой выгрузки сохранять:
  - запрос (параметры)
  - время
  - размер ответа
  - ошибки, если были

## 5. Связка Метрика↔Вебмастер может рваться

**Симптом.** В Вебмастере активна диагностическая ошибка `NO_METRIKA_COUNTER_BINDING`. Данные по кликам/показам по запросам в Вебмастере — нули с 7 апреля, при том что:
- Обход сайта продолжается (есть HTTP_2XX)
- Страницы в поиске обновляются
- Метрика показывает 364 органических визита за тот же период

**Причина.** Привязка счётчика Метрики к Вебмастеру оборвалась. Без неё Вебмастер не может агрегировать часть статистики. При этом органический трафик и обход — идут своим чередом.

**Правило работы.**
- При пропаже данных в Вебмастере первый чек: `NO_METRIKA_COUNTER_BINDING` в диагностике.
- Не интерпретировать нули в Вебмастере как падение трафика без проверки Метрики и самостоятельных сервисов мониторинга позиций.
- Восстановление связки — через интерфейс Метрики и Вебмастера, не автоматически.

## 6. Нормальный срок обновления данных — до 2 недель

**Источник.** Официальная документация Яндекса: `yandex.ru/support/webmaster/ru/diagnosis/possible.html` и смежные разделы.

**Правило работы.**
- Не паниковать, если данные за последние 7-10 дней отсутствуют.
- Пороговое значение для беспокойства — 14 дней без обновления при том, что обход и индексация идут нормально.
- Перед выводами о санкциях убедиться, что нет активных проблем уровня `DNS_ERROR`, `MAIN_PAGE_ERROR`, `SSL_CERTIFICATE_ERROR`, `THREATS`, `URL_ALERT_5XX`.

## 7. Read-only правило применимо ко всем системам

**Системы, где действует read-only по умолчанию:**
- Яндекс Вебмастер
- Яндекс Метрика
- Яндекс Бизнес
- Яндекс Директ

**Запрещено без отдельного прямого согласия владельца:**
- Переобход URL
- Добавление/удаление sitemap
- Изменение региональности
- Изменение настроек счётчика
- Изменение целей
- Изменение доступов
- Любые рекламные изменения
- Удаление счётчиков или связей

**Применимо даже если** метод технически использует `POST`, но только для выгрузки (`Logs API` и подобные) — подтвердить, что метод не меняет настройки.

## 8. Опыт генерации PDF из md-отчётов

**Случай.** Сессия `c0b49890-75bd-41a7-a1bc-ea8239879c70` — конвертация нескольких отчётов AZT в PDF через skill `document-skills/pdf` (weasyprint).

**Что нужно убирать при подготовке md→pdf для внешней передачи:**
- YAML frontmatter (`type`, `tags`, `created`, `status`, `author`, и т.д.)
- Obsidian wiki-ссылки `[[URL|text]]` → обычные ссылки с текстом или markdown `[text](url)`
- Секция "Связанные документы" (wiki-ссылки внутри vault — во внешнем PDF бесполезны)
- Учесть вариации заголовка: `## Связанные документы`, `## 17. Связанные документы`, `## 15. Связанные документы` и т.п. Регулярка должна ловить оба варианта.

**Пример:**
```python
# Секция "Связанные документы" — варианты с номером и без
re.sub(r'\n##\s+(\d+\.\s+)?Связанные документы.*?(?=\n##\s|\Z)', '', md, flags=re.DOTALL)
```

**Правило работы.** Для каждого нового отчёта — просмотреть PDF глазами перед отправкой. Пропущенные wiki-ссылки или frontmatter сразу видны.

## 9. Лимиты MCP-инструментов

**Примеры лимитов, которые мы встречали:**
- `get-report` в Yandex Metrika MCP: `limit` максимум 100. Больше — ошибка валидации.
- `get-popular-pages`, `get-search-phrases`, `get-geography` и др.: максимум 100.
- `get-external-links` в Webmaster MCP: максимум 500.

**Правило работы.**
- При `limit > 100` для Метрики — делать пагинацию через `offset`.
- Перед запуском большого запроса проверять схему инструмента: `parameters.properties.limit.maximum`.

## 10. Количество счётчиков в одном аккаунте

**Кейс.** `get-counters` вернул 5 счётчиков для аккаунта `pavel-lamzin`. Из них 4 — по AZT (1 сайт + 3 карточки), 1 — "Кардио-Уральск" (сторонний счётчик).

**Правило работы.**
- Всегда проверять весь список счётчиков до работы с конкретным.
- Счётчики с пометкой `source: "sprav"` — автоматические от Яндекс Бизнеса, не путать с сайтовыми.
- `owner_login` может быть другим (`teplitsi-azt`, `yndx-metrika-counters-NNN`) — это не мешает delegated-доступу.

## 11. `get_traffic_summary` в Metrika MCP может падать

**Симптом.** Вызов `get_traffic_summary` по счётчику Метрики падал с ошибкой `metrics is not iterable`. Пробовались варианты: без дат, с диапазоном дат, только `date_from`, только `date_to`. Результат одинаковый.

**Причина.** Ошибка на стороне MCP-инструмента или его адаптера. Это не означает, что данных в Метрике нет.

**Правило работы.**
- Не делать вывод "сводка недоступна" или "данных нет" только по падению `get_traffic_summary`.
- Для traffic summary использовать `get_report` или прямой read-only API Метрики `stat/v1/data`.
- Минимальная замена:
  - `metrics=ym:s:visits,ym:s:users,ym:s:pageviews,ym:s:bounceRate,ym:s:avgVisitDurationSeconds`
  - без `dimensions` для общей сводки или `dimensions=ym:s:date` для динамики.
- Ошибку инструмента фиксировать в выгрузке/комментарии, но не смешивать с качеством данных Метрики.

## 12. MCP Метрики удобен для точечных срезов, но для полной выгрузки лучше direct API

**Симптом.** Нужно было выкачать "вообще всё, что можно" по каждому счётчику. Через MCP это неудобно: лимиты, ручные вызовы, риск потерять часть отчётов и сложнее сохранять сырые ответы.

**Решение.** Для полной выгрузки использовался read-only API Метрики напрямую с OAuth-токеном из `.mcp.json`, с сохранением JSON в `_data/`.

**Правило работы.**
- MCP использовать для быстрой проверки гипотез и небольших срезов.
- Для полной воспроизводимой выгрузки использовать direct API:
  - сохранять `request`;
  - сохранять `meta`;
  - сохранять `rows`;
  - сохранять ошибки API внутри соответствующего блока;
  - фиксировать дату выгрузки и период.
- Прямой API допустим только для read-only методов. Любые методы, меняющие настройки, запрещены без отдельного согласия.

## 13. `ym:s:pageURL` — неправильное поле для популярных просмотренных страниц

**Симптом.** Запрос популярные страницы через `dimensions=ym:s:pageURL` вернул ошибку:

```text
Incorrectly specified attribute, value: ym:s:pageURL
```

**Причина.** Просмотренные URL относятся к hit/pageview-набору Метрики, а не к session-набору.

**Правило работы.**
- Для страниц входа использовать `ym:s:startURL`.
- Для просмотренных URL использовать `ym:pv:URL`.
- Для просмотров использовать `ym:pv:pageviews`.
- Для пользователей по просмотренным URL использовать `ym:pv:users`.
- Не считать `ym:s:startURL` и `ym:pv:URL` взаимозаменяемыми:
  - `startURL` — первая страница визита;
  - `URL` — любой просмотренный URL внутри визита.

**Пример правильного запроса для популярных страниц:**

```text
dimensions=ym:pv:URL
metrics=ym:pv:pageviews,ym:pv:users
sort=-ym:pv:pageviews
```

## 14. Goal-метрики Метрики нужно выгружать чанками

**Симптом.** Попытка выгрузить все goal-метрики одним запросом вернула ошибку:

```text
Exceeded number of metrics in request, value: 76, limit: 20
```

**Причина.** У API Метрики есть лимит на количество метрик в одном запросе. У AZT много целей, особенно у счётчиков Яндекс Бизнеса.

**Правило работы.**
- Goal-метрики разбивать на чанки.
- В каждом chunk фиксировать:
  - список goal ID;
  - список метрик;
  - dimensions;
  - период;
  - статус запроса;
  - ошибки, если были.
- Выгрузка целей считается неполной, если хотя бы один chunk завершился ошибкой.
- Нельзя делать вывод по целям, если неизвестно, все ли важные цели попали в выгрузку.

## 15. `vacuum`-отчёты — обязательный слой для Яндекс Бизнеса

**Симптом.** В кабинете Яндекс Бизнеса видны действия в карточках: показы, переходы на сайт, звонки, маршруты. В обычных goal-срезах Метрики часть этих данных не видна.

**Причина.** Часть действий карточки доступна в отчёте Метрики `Целевые события на Яндексе`, а не только как цели счётчика.

**Рабочие поля:**
- `ym:s:vacuumSurface` — поверхность Яндекса: Поиск, Карты, мобильные Карты, Навигатор и т.д.
- `ym:s:vacuumOrganization` — организация или адрес.
- `ym:s:vacuumEvent` — действие в карточке.
- `ym:s:vacuumevents` — количество событий.
- `ym:s:visits` — визиты.
- `ym:s:vacuumusers` — пользователи.
- `ym:s:vacuumeventsPerUser` — событий на пользователя.

**Правило работы.**
- Для счётчиков `source: "sprav"` обязательно выгружать `vacuum`-срезы.
- Основные срезы:
  - `vacuumEvent`;
  - `vacuumSurface`;
  - `vacuumSurface, vacuumOrganization, vacuumEvent`;
  - `date, vacuumEvent`;
  - `searchPhrase, vacuumSurface, vacuumEvent`;
  - `searchPhrase, vacuumOrganization, vacuumEvent`.
- Показы карточки и профиля — верх воронки, не лиды.
- Переходы на сайт, звонки и маршруты — коммерчески значимые действия, но не продажи без дополнительного подтверждения.

## 16. `vacuum` не применим к обычному счётчику сайта

**Симптом.** При добавлении `vacuum`-отчётов во все JSON-файлы счётчиков обычный счётчик сайта `87177110` вернул ошибки по этим срезам.

**Причина.** `vacuum`-отчёты относятся к автоматическим счётчикам организаций Яндекс Бизнеса, а не к обычному счётчику сайта `azt-teplica.ru`.

**Правило работы.**
- Для счётчика сайта `87177110` блок Яндекс Бизнеса помечать как `not_applicable`.
- Для счётчиков организаций (`source: "sprav"`) выгружать `vacuum`.
- Не считать отсутствие `vacuum` на сайтовом счётчике ошибкой данных.

## 17. Поисковые фразы можно совмещать с событиями Яндекс Бизнеса

**Симптом.** В кабинете Яндекс Бизнеса есть вкладка "Как вас находят". Нужно было получить не только действия, но и запросы, по которым находили карточку.

**Решение.** Рабочий срез через Метрику:

```text
dimensions=ym:s:searchPhrase,ym:s:vacuumSurface,ym:s:vacuumEvent
metrics=ym:s:vacuumevents,ym:s:visits,ym:s:vacuumusers
sort=-ym:s:vacuumevents
```

**Правило работы.**
- Для карточек Яндекс Бизнеса поисковые фразы нужно анализировать отдельно от поисковых фраз сайта.
- Фраза карточки показывает спрос внутри сервисов Яндекса, а не обязательно переход на сайт.
- Эти данные полезны для:
  - региональной модели;
  - карточек;
  - SEO/GEO;
  - объявлений;
  - вопросов агентству.

## 18. Поведенческие метрики счётчиков Яндекс Бизнеса не трактовать как поведение на сайте

**Симптом.** В API по счётчикам организаций доступны обычные метрики вроде отказов, глубины и времени.

**Причина.** Автоматический счётчик организации работает на поверхностях Яндекса (`yandex.ru/maps`, приложения и т.д.), а не на сайте AZT.

**Правило работы.**
- Для счётчиков Яндекс Бизнеса не делать выводы по:
  - bounceRate;
  - pageDepth;
  - avgVisitDurationSeconds;
  - поведению на URL сайта.
- Использовать:
  - `vacuumevents`;
  - `vacuumusers`;
  - `vacuumSurface`;
  - `vacuumEvent`;
  - цели карточки;
  - переходы на сайт;
  - звонки;
  - маршруты.

## 19. Большие JSON выгрузки Метрики и Вебмастера должны идти через Git LFS

**Кейс.** Полный snapshot Яндекс Вебмастера весил около 160 MB. Полная выгрузка основного счётчика Метрики — десятки MB.

**Правило работы.**
- Для больших данных использовать Git LFS.
- Текущие правила:
  - `10 Projects/AZT/_data/**/*.json`
  - `10 Projects/AZT/_reports/**/*.pdf`
- После добавления больших файлов проверять:
  - `git check-attr filter diff merge -- <file>`;
  - `git lfs fsck`;
  - `git lfs ls-files --size`.
- В staged-объекте должен быть LFS pointer, а не полный JSON.

## 20. Ошибки API нужно хранить как данные, но потом классифицировать

**Симптом.** При первичной выгрузке в JSON попали блоки с ошибками API: неправильный `ym:s:pageURL`, превышение лимита goal-метрик, `vacuum` на обычном счётчике сайта.

**Правило работы.**
- Ошибки API не удалять молча.
- Сначала сохранить и понять причину.
- Затем:
  - исправить запрос, если он был неправильным;
  - разбить запрос на чанки, если превышен лимит;
  - пометить блок как `not_applicable`, если отчёт неприменим к счётчику.
- После исправления выполнить повторную проверку, что `report_errors: none`.

## 21. Сырые данные и агрегированные отчёты — разные уровни

**Симптом.** Возник вопрос: являются ли выгруженные данные Метрики сырыми.

**Ответ.** Текущие выгрузки через `stat/v1/data` — агрегированные отчёты по dimensions/metrics. Это достаточно для большинства управленческих и SEO/маркетинговых выводов, но это не уровень "каждый визит/хит отдельной строкой".

**Правило работы.**
- Агрегированные отчёты использовать для:
  - источников;
  - посадочных;
  - регионов;
  - устройств;
  - целей;
  - UTM;
  - карточек Яндекс Бизнеса.
- Сырые данные уровня визитов и хитов требуют Logs API.
- Logs API использовать только если агрегированные срезы не отвечают на вопрос.
- Перед Logs API отдельно подтверждать, что метод не меняет настройки проекта.

## 22. Итоговый анализ строится не "ещё одним отчётом", а наслаиванием источников

**Урок.** Новые данные Метрики, Яндекс Бизнеса, Wordstat, Вебвизора, Директа, конкурентов и планов агентства не должны хаотично переписывать старые документы.

**Правило работы.**
- Сначала сохранить данные в `_data`.
- Потом проверить качество.
- Потом классифицировать данные:
  - технический риск;
  - спрос;
  - трафик;
  - лидовая цель;
  - коммерческое действие;
  - верх воронки;
  - UX-наблюдение;
  - рекламная гипотеза.
- Потом обновить матрицу доказательств.
- Потом обновить карту потерь и возможностей.
- Потом обновить приоритеты.
- Только после этого менять отчёт собственнику, ТЗ разработчику или вопросы агентству.

**Итоговая цель.** Не сделать красивый набор документов, а построить управляемую коммерческую модель онлайн-присутствия через сайт, где каждый рубль, задача и вывод привязаны к доказанному спросу, технической готовности и измеримому коммерческому результату.
