# AZT — Регламент анализа рекламы и планов агентства

## Назначение

Регламент описывает, как проверять медиаплан, семантику, тексты объявлений, план SEO, план Директа, план разработки сайта, прогресс агентства и ТЗ на SEO-статьи.

Задача не в том, чтобы согласиться или спорить с агентством, а в том, чтобы проверить планы по данным и встроить их в общую модель роста AZT.

## Какие документы агентства анализируются

Входные документы:

- медиаплан;
- семантика;
- тексты объявлений;
- структура рекламных кампаний;
- план по Яндекс Директу;
- SEO-план;
- план разработки сайта;
- текущий прогресс работ;
- ТЗ на SEO-статьи;
- контент-план;
- отчеты агентства;
- комментарии по выполненным задачам.

## Общий принцип проверки

Каждый пункт плана агентства проверяется по схеме:

```text
Что предлагают
→ на основании каких данных
→ какую страницу или канал затрагивает
→ есть ли технические риски
→ какой KPI
→ как проверяется выполнение
→ что делать до запуска
→ что делать после запуска
```

## Медиаплан

Для медиаплана проверяется:

- какие регионы покрываются;
- какие продукты продвигаются;
- какие посадочные используются;
- какие бюджеты и сроки;
- какие цели считаются результатом;
- есть ли UTM-разметка;
- нет ли трафика на проблемные URL;
- не противоречит ли медиаплан техническому состоянию сайта;
- какие гипотезы медиаплан проверяет;
- какие выводы будут сделаны по итогам теста.

## Семантика

Семантика проверяется по источникам:

- Wordstat;
- Вебмастер;
- Метрика;
- Яндекс Бизнес;
- Директ;
- конкуренты;
- текущая структура сайта.

Каждый кластер получает назначение:

- SEO-страница;
- региональная посадочная;
- карточка товара;
- статья;
- рекламная группа;
- карточка Яндекс Бизнеса;
- не использовать сейчас.

## Тексты объявлений

Для объявлений проверяется:

- соответствие запросу;
- соответствие посадочной;
- соответствие региону;
- наличие реального УТП;
- отсутствие обещаний, которых нет на сайте;
- согласованность с ценой, доставкой, монтажом, рассрочкой;
- наличие UTM;
- риск вести пользователя на слабую страницу.

## SEO-план и ТЗ на статьи

SEO-статьи согласуются только после проверки:

- есть ли спрос по теме;
- информационный или коммерческий это спрос;
- есть ли уже страница, которая должна закрывать этот запрос;
- не конкурирует ли статья с коммерческой посадочной;
- индексируется ли раздел;
- нет ли технических проблем шаблона;
- есть ли место статьи в воронке;
- какой следующий шаг должен сделать пользователь после чтения;
- как будет измеряться результат.

Нельзя согласовывать статьи только потому, что тема выглядит полезной.

## План разработки сайта

План разработки сопоставляется с:

- P0/P1/P2 задачами;
- Вебмастером;
- Метрикой;
- техническим ТЗ;
- рекламными посадочными;
- SEO/GEO-планом;
- UX-наблюдениями из Вебвизора.

Если агентство предлагает улучшения до исправления P0-техники, нужно зафиксировать зависимость:

```text
Задача полезна, но запускать ее до исправления sitemap/canonical/404 рискованно, потому что новые страницы или рекламный трафик могут попасть в технически проблемную среду.
```

## Текущий прогресс агентства

Прогресс оценивается не по факту "сделано/не сделано", а по критериям:

- задача была в плане;
- задача выполнена;
- есть подтверждение выполнения;
- результат проверен независимо;
- эффект виден или еще не мог проявиться;
- есть открытые риски;
- есть следующий шаг.

## Вопросы агентству

Вопросы формулируются как:

```text
Факт из данных → вопрос → ожидаемое подтверждение
```

Пример:

```text
Вебмастер фиксирует проблемы с региональными URL, а медиаплан включает региональный трафик. Какие посадочные будут использоваться и как проверяется их техническая готовность перед запуском?
```

## Выходные артефакты

После анализа агентских документов обновляются:

- матрица доказательств;
- карта потерь и возможностей;
- список вопросов агентству;
- приоритеты P0/P1/P2;
- план развития;
- ТЗ разработчику;
- отчет собственнику, если вывод влияет на управленческое решение.
