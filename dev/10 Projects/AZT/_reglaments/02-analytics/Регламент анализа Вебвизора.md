# AZT — Регламент анализа Вебвизора

## Назначение

Вебвизор используется для качественной диагностики поведения пользователей. Он не заменяет Метрику, Вебмастер или данные конверсий, а объясняет, почему пользователи могут не доходить до заявки, звонка, выбора товара или другого коммерческого действия.

Вебвизор отвечает на вопрос:

```text
Что пользователь реально делал на странице и где возникло препятствие?
```

## Ограничение

Вебвизор не является статистическим доказательством сам по себе.

Нельзя писать:

```text
80% пользователей не находят кнопку, если просмотрено 10 записей.
```

Корректно:

```text
В просмотренной выборке повторяется паттерн: пользователи на мобильных устройствах доходят до блока выбора теплицы, но не продолжают сценарий заявки.
```

## Место в модели

Вебвизор включается после Метрики.

Правильная последовательность:

1. Метрика находит проблемный сегмент.
2. По этому сегменту отбираются записи Вебвизора.
3. Записи просматриваются по единой схеме.
4. Повторяющиеся паттерны фиксируются как UX-наблюдения.
5. Наблюдения попадают в матрицу доказательств только с пометкой `качественное подтверждение`.

## Приоритетные выборки

Сначала смотреть не все записи подряд, а целевые сегменты:

- мобильные пользователи на ключевых посадочных без цели;
- пользователи платного трафика без цели;
- пользователи из регионов Пермь, Тюмень, Екатеринбург, Ижевск;
- переходы из Яндекс Бизнеса на сайт;
- визиты на проблемные URL из Вебмастера;
- визиты с открытием формы, но без отправки;
- визиты с длительным временем на сайте, но без цели;
- визиты с возвратами к цене, доставке, монтажу, гарантиям;
- визиты, где пользователь активно кликает, но не достигает результата.

## Карточка просмотра записи

Для каждой записи фиксируется:

- дата и время;
- счетчик;
- источник;
- устройство;
- город;
- посадочная страница;
- достигнутые цели;
- длительность;
- глубина;
- ключевой сценарий;
- наблюдаемая проблема;
- гипотеза причины;
- доказательность: единичное наблюдение или повторяющийся паттерн;
- ссылка/ID записи, если доступно.

## Справочник UX-проблем

Проблемы кодируются единообразно:

- `не видит CTA`;
- `не понимает цену`;
- `не находит доставку`;
- `не находит монтаж`;
- `не доверяет офферу`;
- `форма неочевидна`;
- `форма не работает`;
- `клик по неактивному элементу`;
- `мобильная верстка мешает`;
- `долго выбирает модель`;
- `возврат к карточке товара`;
- `уход после цены`;
- `уход после доставки`;
- `уход после контактов`;
- `переход между регионами`;
- `техническая ошибка`.

## Связь с выводами

Вебвизор может:

- усилить вывод Метрики;
- объяснить низкую конверсию посадочной;
- подтвердить UX-проблему;
- показать, что техническая ошибка реально затрагивает пользователя;
- дать задачу на улучшение посадочной.

Вебвизор не может без дополнительных данных:

- доказать общий процент потерь;
- доказать финансовые потери;
- заменить анализ целей;
- заменить проверку сайта;
- заменить конкурентный анализ.

## Выходные артефакты

По итогам анализа Вебвизора создаются:

- список повторяющихся UX-паттернов;
- связка паттерн → сегмент → страница → источник;
- задачи для сайта или посадочной;
- вопросы к агентству, если проблема связана с рекламным трафиком;
- уточнения в матрицу доказательств.
