# AZT — Регламент матрицы доказательств

## Назначение

Матрица доказательств нужна для того, чтобы каждый важный вывод по проекту AZT был связан с источниками, цифрами, статусом доказанности, бизнес-влиянием и конкретным действием.

Это центральный рабочий инструмент. Отчеты для собственника, ТЗ разработчику и вопросы агентству должны опираться на матрицу, а не на набор разрозненных наблюдений.

## Когда создавать матрицу

Матрица создается или обновляется после появления нового значимого источника:

- выгрузка Яндекс Вебмастера;
- выгрузка Яндекс Метрики;
- выгрузка счетчиков Яндекс Бизнеса;
- данные Яндекс Директа;
- проверка сайта, sitemap, robots, canonical, schema;
- аудит агентства;
- конкурентный анализ;
- CRM или реальные заявки, если появятся.

## Обязательные поля

Каждая строка матрицы должна содержать:

- `ID`;
- `Тезис`;
- `Категория`;
- `Источник 1`;
- `Источник 2`;
- `Источник 3`;
- `Цифры`;
- `Статус доказанности`;
- `Бизнес-влияние`;
- `Риск бездействия`;
- `Приоритет`;
- `Рекомендуемое действие`;
- `Критерий проверки`;
- `Куда попадает вывод`;
- `Открытые вопросы`.

## ID

Формат ID:

```text
AZT-[SOURCE]-[AREA]-[NNN]
```

Примеры:

- `AZT-WM-INDEX-001` — индексирование из Вебмастера;
- `AZT-WM-REG-001` — региональная проблема из Вебмастера;
- `AZT-MET-CONV-001` — конверсии из Метрики;
- `AZT-YB-CARD-001` — карточки Яндекс Бизнеса;
- `AZT-TECH-SITEMAP-001` — техническая проблема сайта;
- `AZT-AGENCY-PLAN-001` — проверка пункта агентства.

ID не меняется после создания строки. Если вывод уточнился, обновляется статус и комментарий, но не ID.

## Категории

Допустимые категории:

- `индексация`;
- `техническая ошибка`;
- `региональность`;
- `посадочная страница`;
- `органический спрос`;
- `платный трафик`;
- `Яндекс Бизнес`;
- `цели и конверсии`;
- `контент`;
- `агентство`;
- `конкуренты`;
- `данные требуют проверки`.

## Статусы доказанности

Используются только фиксированные статусы:

- `доказано`;
- `подтверждено технически`;
- `усилено Метрикой`;
- `вероятно`;
- `требует проверки`;
- `не подтвердилось`;
- `не анализировалось`.

Статус `доказано` допустим только при наличии факта и подтверждения его значимости. Например: Вебмастер показывает исключение URL, а Метрика показывает трафик или бизнес-действия на связанном участке.

## Цифры

В поле `Цифры` пишутся только проверенные значения:

- период;
- источник;
- значение;
- единица измерения;
- при необходимости URL или ID счетчика.

Пример:

```text
Вебмастер, 2026-04-16: 428 битых внутренних ссылок.
Метрика, счетчик 87177110, 2026-03-15..2026-04-16: /catalog/catalog/teplicy/ — 907 просмотров, 50 входов.
```

Нельзя писать:

```text
Много ошибок.
Плохая конверсия.
Скорее всего теряется трафик.
```

## Бизнес-влияние

Бизнес-влияние формулируется через управленческий смысл:

- риск потери индексации;
- риск потери регионального спроса;
- потеря переходов на сайт;
- потеря звонков;
- потеря маршрутов;
- расход бюджета на слабую посадочную;
- искажение аналитики;
- невозможность оценить эффективность агентства.

Если бизнес-влияние не доказано, пишется:

```text
Бизнес-влияние не оценено. Требуется срез Метрики / Директа / CRM.
```

## Приоритет

Приоритет определяется по связке:

```text
Impact + Risk + Evidence + Effort + Time-to-effect
```

P0 ставится только если:

- есть высокий технический или коммерческий риск;
- проблема подтверждена данными;
- без исправления невозможно корректно двигать SEO, рекламу или аналитику.

P1 ставится, если проблема важна, но не блокирует индексирование или коммерческие действия прямо сейчас.

P2 ставится для улучшений, гипотез, контента и задач без срочного риска.

## Куда попадает вывод

В поле `Куда попадает вывод` указывается, где вывод должен быть использован:

- `отчет собственнику`;
- `ТЗ разработчику`;
- `вопрос агентству`;
- `план развития`;
- `контент-план`;
- `рекламный аудит`;
- `требует дополнительной проверки`.

Один вывод может попадать в несколько документов, но формулировка должна адаптироваться под адресата.

## Пример строки

```text
ID: AZT-WM-REG-001
Тезис: региональные страницы работают нестабильно и часть регионов находится в зоне риска.
Категория: региональность
Источник 1: Вебмастер — /izh/ исключен как NOT_CANONICAL.
Источник 2: Вебмастер — /prm/ имеет LOW_QUALITY.
Источник 3: Метрика/Яндекс Бизнес — региональные карточки дают показы, переходы, звонки и маршруты.
Цифры: /izh/ — 14 NOT_CANONICAL; /prm/ — 6 LOW_QUALITY; Пермь, Яндекс Бизнес — 50 623 события за период счетчика.
Статус доказанности: доказано частично, требуется связка с посадочными сайта.
Бизнес-влияние: риск потери регионального органического спроса и недоиспользования карточек.
Риск бездействия: следующие региональные страницы могут получить похожие сигналы.
Приоритет: P0/P1 после проверки шаблона регионов.
Рекомендуемое действие: проверить canonical, региональные шаблоны, sitemap и посадочные для всех регионов.
Критерий проверки: повторная выгрузка Вебмастера и Метрики после исправлений.
Куда попадает вывод: ТЗ разработчику, отчет собственнику, вопросы агентству.
Открытые вопросы: кто отвечает за региональные посадочные и карточки.
```

## Правила работы

Матрица обновляется до переписывания финальных документов.

Нельзя переносить вывод в отчет собственнику, если в матрице нет источника и статуса.

Нельзя удалять строку молча. Если вывод больше не актуален, статус меняется на `не подтвердилось` или `закрыто`, а причина фиксируется.

Если новая информация меняет приоритет, в строке указывается, какая именно цифра изменила решение.
