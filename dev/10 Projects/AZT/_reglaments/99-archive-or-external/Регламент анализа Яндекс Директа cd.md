# Регламент анализа Яндекс Директа

Статус: **черновик-заготовка**. Заполняется после получения доступа к рекламному кабинету и первой реальной работы с данными Директа. До этого документ — список вопросов, которые надо закрыть, и предполагаемая структура анализа.

## Назначение

Регламент описывает работу с данными Яндекс Директа: выгрузку, проверку качества, анализ эффективности и связку с Метрикой и Вебмастером.

Цель — оценивать эффективность не как "сколько потрачено", а как "какой коммерческий результат получен на каждую посадочную/сегмент/регион и что мешает масштабировать прибыльные кампании".

## Read-only правило

Для Яндекс Директа действует такое же read-only правило, как для Вебмастера, Метрики и Яндекс Бизнеса (см. [[Регламент выгрузки и качества данных]]).

**Запрещено без прямого согласия:**
- создавать или удалять кампании;
- менять ставки, бюджеты, таргетинги;
- останавливать или запускать кампании и объявления;
- добавлять или удалять минус-слова, ключи, аудитории;
- менять сегменты ретаргетинга;
- менять связку с Метрикой.

Любые изменения — только через владельца по текстовому описанию.

## Источники данных

Когда доступ появится, использовать:
- веб-интерфейс Директа — для ручной проверки структуры кампаний и слов-проблем;
- API Директа — для регулярной выгрузки метрик;
- Метрика — для связки `рекламный визит → посадочная → цель`;
- отчёт Метрики "Директ — сводка" и "Директ — расходы";
- UTM-метки для связки кампания → посадочная → цель в отчётах Метрики.

## Что нужно выяснить до первого полного анализа

Перед тем как делать выводы, получить ответы:

1. Какие цели Метрики считаются лидовыми для Директа? Лидовые vs микроконверсии (см. [[Регламент целей и конверсий]]).
2. Используется ли Callibri или другой коллтрекинг для учёта звонков в Директе?
3. Настроена ли интеграция "Контекст//Динамика" (ID 87092) — в Метрике цели этой интеграции уже есть.
4. Какая модель атрибуции считается основной: `Последний значимый переход`, `Последний переход`, `Первый переход`?
5. Есть ли доступ к отчёту расходов по кампаниям в Метрике?
6. Какой период считается контрольным — для дневного, недельного, месячного ритма.

## Обязательные срезы

Минимальный набор, который снимается при baseline:

- `кампания → визиты → цели → CPL`;
- `кампания → посадочная → устройство`;
- `кампания → регион пользователя → посадочная`;
- `кампания → ключевая фраза → посадочная → цель` (для поисковых);
- `кампания → площадка (РСЯ) → посадочная → цель`;
- `рекламный визит → вебвизор → поведение на посадочной`;
- `эффективность объявлений → CTR → CPC → конверсия в цель`;
- `минус-слова → пропущенные нецелевые запросы`.

Дополнительно для контроля:
- рекламные URL с ошибками 404, редиректами, `erid` → в Метрике и Вебмастере;
- UTM на посадочных, которых нет в sitemap;
- UTM на посадочных с `LOW_QUALITY` / `NOT_CANONICAL` в Вебмастере.

## Экономика лида

Для каждой кампании фиксировать:
- расход за период;
- визиты;
- лидовые цели (формы, звонки Callibri, клики по телефону);
- коммерческие действия (переход в мессенджер, начало заказа);
- средний CPL;
- средний CPC;
- долю отказов;
- конверсию в заявку.

Для каждой посадочной страницы фиксировать:
- расход, который пришёл на страницу;
- визиты;
- достижения лидовых целей;
- CPL страницы;
- не попала ли страница в `HTTP_ERROR` / `NOT_CANONICAL` / `LOW_QUALITY` в Вебмастере.

Если посадочная даёт визиты за деньги и имеет технические проблемы — это приоритет на исправление.

## Сегментация кампаний

До масштабирования проверить, что кампании разделены по осмысленным сегментам:
- поиск vs сети vs ретаргетинг;
- брендовые vs коммерческие vs информационные;
- регионы (Пермь, Екатеринбург, Тюмень, Ижевск);
- категории (теплицы, парники, беседки, поликарбонат, грядки);
- спецпредложения (рассрочка, хранение, доставка) — отдельными кампаниями.

Если кампании перемешаны, сначала выводы делать с оговоркой — без декомпозиции нельзя точно понять, что работает.

## Связь с Метрикой

Перед анализом Директа проверить:
- связка счётчик Метрики (87177110) ↔ кабинет Директа активна;
- цели, которые должны считаться конверсиями, отмечены как "ключевые" и попадают в отчёты Директа;
- UTM-метки реальных посадочных соответствуют ожидаемым в Директе;
- нет ли пропущенных меток.

## Связь с Вебмастером

Для каждой посадочной, на которую тратится реклама, проверить в Вебмастере:
- индексируется ли она;
- нет ли `NOT_CANONICAL`;
- нет ли `HTTP_ERROR`;
- нет ли `LOW_QUALITY`;
- попадает ли в sitemap.

Расход на страницу с техническими проблемами → задача P0 на техническую часть до масштабирования рекламы.

## Связь с Яндекс Бизнесом

Для регионов проверить:
- есть ли у региона карточка Яндекс Бизнеса;
- сколько действий в карточке и какие (звонки, маршруты, переходы);
- пересекается ли рекламная семантика региона с поисковыми фразами карточки;
- дешевле ли получать действие через карточку, чем через Директ (при прочих равных).

Это помогает понять, где Директ покупает то, что уже есть в Яндекс Бизнесе бесплатно.

## Вопросы, которые агентство обязано закрыть

Эти вопросы — из [[Регламент вопросов агентству]] применительно к Директу:

- На каких данных основана текущая структура кампаний?
- Какой CPL считается целевым и по каким данным он выставлен?
- Как выбраны приоритетные посадочные?
- Как учтены технические проблемы сайта в выборе посадочных?
- Какие кампании масштабируются и на основании какого KPI?
- Какая модель атрибуции используется для расчёта эффективности?
- Как сегментированы брендовые vs коммерческие запросы?
- Есть ли отдельная ретаргетинговая стратегия и какие сегменты используются?
- Как проверяется доступность рекламных URL перед запуском объявлений?

## Типовые проблемы для проверки

Список симптомов, которые при появлении доступа надо проверить в первую очередь:

- URL с `?erid=...` возвращают 404 в Вебмастере (уже видели в Search Events Samples);
- утечка рекламного трафика на региональные страницы, которые исключаются как `NOT_CANONICAL`;
- расход на страницы, которые Яндекс исключил как `LOW_QUALITY`;
- брендовый трафик отдельно не выделен → переплата за то, что получили бы бесплатно;
- РСЯ-площадки без чистки с низким качеством визитов (высокий bounce, 0 целей);
- пересечение семантики Директа и органических брендовых запросов без защитного решения;
- отсутствие ретаргетинга по посадочным с высокой стоимостью.

## Формат baseline-снимка

При первой полной выгрузке Директа сформировать документ `<project> — Baseline Яндекс Директ YYYY-MM-DD.md` со структурой:

- Счета и кампании (ID, статус, даты запуска).
- Общая сводка: расход, визиты, цели, CPL за период.
- Сегменты: поиск/сети/ретаргетинг, регионы, категории.
- Топ кампаний по CPL.
- Проблемные URL (404, NOT_CANONICAL, LOW_QUALITY) с расходом.
- Цели Метрики, связанные с Директом, и их достижения.
- Открытые вопросы.

## Связь с матрицей доказательств

Каждый вывод по Директу фиксируется в матрице доказательств с категорией `платный трафик` и источниками `Директ API`, `Метрика`, `Вебмастер`, `Яндекс Бизнес`.

## Что делать после первой работы

После первой полной выгрузки:
1. Снять этот статус `draft` и перевести в `final`.
2. Дополнить примерами реальных запросов и ответов API.
3. Указать реальные поля экспорта, которые пригодились.
4. Зафиксировать найденные проблемы и методы их обнаружения.
