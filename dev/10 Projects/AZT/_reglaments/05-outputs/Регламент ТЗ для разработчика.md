# Регламент ТЗ для разработчика

## Назначение

ТЗ для разработчика должно превращать аналитический вывод в исполнимую техническую задачу.

В ТЗ не должно быть управленческой риторики. Разработчику нужны конкретные URL, правила, ожидаемое поведение и критерии приемки.

## Структура задачи

Каждая задача описывается по шаблону:

```text
ID:
Приоритет:
Проблема:
Где проявляется:
Примеры:
Что сделать:
Как должно работать:
Критерий готовности:
Как проверить:
Источник:
```

## Приоритет

Приоритет берется из матрицы доказательств.

P0:

- мешает индексированию;
- генерирует битую ссылочную массу;
- создает 404 или неправильные canonical;
- затрагивает региональные страницы в зоне риска;
- влияет на рекламные посадочные или коммерческие цели.

P1:

- важно для SEO/GEO, но не блокирует работу прямо сейчас;
- влияет на качество страниц;
- требует исправления до масштабирования контента или рекламы.

P2:

- улучшения;
- рефакторинг;
- косметика;
- долгосрочные задачи.

## Примеры

Для URL-проблем обязательно давать конкретные URL.

Для sitemap:

```text
Как сейчас:
sitemap содержит URL, которые отдают 404 или ведут на неправильные разделы.

Как должно быть:
sitemap содержит только канонические индексируемые URL с кодом 200.
```

Для canonical:

```text
Как сейчас:
региональная страница указывает canonical на другую страницу или конфликтует с собственной региональностью.

Как должно быть:
канонический URL соответствует индексируемой региональной странице.
```

Для schema:

```text
Как сейчас:
разметка содержит неполные или неподходящие поля.

Как должно быть:
разметка соответствует типу страницы и не противоречит видимому содержимому.
```

## Acceptance criteria

Критерии готовности должны быть проверяемыми:

- URL отдает `200`;
- URL отсутствует в списке битых ссылок;
- canonical указывает на корректный URL;
- sitemap не содержит дублей и 404;
- страница доступна мобильному роботу;
- если URL должен быть удален из индекса, он отдает корректный `301`, `404` или `410` по согласованной логике;
- robots.txt не блокирует нужные индексируемые страницы;
- тестовая выборка URL проходит проверку сразу после исправления;
- Вебмастер перестает фиксировать ошибку после переобхода;
- Метрика не получает новый трафик на мусорный URL после контрольного периода.

## Запреты

В ТЗ нельзя писать:

- "улучшить SEO";
- "навести порядок";
- "поправить сайт";
- "сделать красиво";
- "разобраться с регионами".

Нужно писать конкретно:

```text
Для всех региональных разделов проверить canonical. Если страница является самостоятельной региональной посадочной, canonical должен указывать на саму себя. Список проверяемых URL приложен.
```

## Связь с отчетом бизнесу

Отчет бизнесу объясняет, почему задача важна.

ТЗ разработчику объясняет, что именно сделать и как принять работу.

Один и тот же вывод должен иметь разные формулировки в этих документах.
