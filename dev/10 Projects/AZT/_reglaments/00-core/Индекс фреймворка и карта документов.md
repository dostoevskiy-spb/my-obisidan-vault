# AZT — Индекс фреймворка и карта документов

## Назначение

Документ является навигационной картой фреймворка AZT.

Он отвечает на вопросы:

- где лежит нужный документ;
- зачем он нужен;
- на каком этапе используется;
- является ли документ регламентом, рабочим журналом, аналитикой или финальным выходом;
- какие документы нельзя использовать отдельно от доказательной базы.

## Главная точка входа

Основная точка входа:

- [[Сквозной флоу управляемой системы роста]]

Если непонятно, с чего начинать, нужно открывать сквозной флоу. Этот индекс используется как карта документов и быстрый справочник.

## Карта папок

| Папка | Назначение | Что хранится |
|---|---|---|
| `01-audit` | доказательные анализы и аудиты | анализ Вебмастера, Метрики, источников, технические выводы |
| `02-owner` | документы для собственника | краткие и итоговые управленческие отчеты |
| `03-dev` | документы для разработки | ТЗ, P0/P1/P2, критерии приемки |
| `04-strategy` | стратегия и развитие | план развития, коммерческая модель, каналы, приоритеты |
| `05-agency` | взаимодействие с агентством | планы агентства, вопросы, сверки, замечания |
| `06-internal` | внутреннее управление | время, решения, действия, изменения, прогресс, влияние |
| `_baseline` | исходные снимки | baseline до изменений |
| `_data` | сырые и агрегированные данные | JSON, выгрузки API, таблицы |
| `_reports` | внешние отчеты и тяжелые файлы | PDF, DOCX, XLSX при необходимости |
| `_reglaments` | правила работы | флоу, регламенты, шаблоны, методология |

## Типы документов

| Тип | Назначение | Можно ли использовать для решений |
|---|---|---|
| Регламент | задает правила работы | да, как процессный источник |
| Сырые данные | фиксируют исходные данные | нет без проверки качества |
| Baseline | фиксирует состояние до изменений | да, для сравнения после внедрения |
| Первичный анализ | разбирает один источник | ограниченно, только в рамках источника |
| Матрица доказательств | объединяет факты и статусы | да |
| Карта потерь/возможностей | переводит факты в бизнес-смысл | да, после матрицы |
| Карта решений | фиксирует управленческий выбор | да, если есть основание |
| ТЗ | передает задачи исполнителю | да, если есть критерии приемки |
| Отчет собственнику | управленческое резюме | да, если основан на матрице и модели |
| Журнал изменений | фиксирует внедрения | да, для контроля |
| Карта влияния | оценивает эффект изменений | да, с учетом уверенности и ограничений |

## Правило

Документ не должен "лежать рядом". Каждый значимый документ должен быть связан минимум с одним этапом флоу, одним входным источником и одним следующим действием.
