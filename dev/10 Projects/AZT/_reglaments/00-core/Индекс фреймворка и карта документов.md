# Индекс фреймворка и карта документов

## Назначение

Документ является навигационной картой фреймворка.

Он отвечает на вопросы:

- где лежит нужный документ;
- зачем он нужен;
- на каком этапе используется;
- является ли документ регламентом, рабочим журналом, аналитикой или финальным выходом;
- какие документы нельзя использовать отдельно от доказательной базы.

## Поля статуса документов

Каждый документ фреймворка имеет два поля статуса с разным смыслом:

| Поле | Что означает | Возможные значения |
|---|---|---|
| `status` | текстовая готовность (можно ли читать и использовать как документ) | `draft` / `in-progress` / `ready` |
| `evidence_status` | доказательная зрелость (подкреплены ли выводы проверенными данными) | `draft` / `checked` / `final` |

**Правило:** Gates (`[[QA Gate контроль качества]]`) проверяют `evidence_status`, не `status`. Документ может быть `status: ready` (текстово завершён) и при этом `evidence_status: draft` (данные ещё не проверены).

## Главная точка входа

Основная точка входа:

- [[Сквозной флоу управляемой системы роста]]

Если непонятно, с чего начинать, нужно открывать сквозной флоу. Этот индекс используется как карта документов и быстрый справочник.

## Типы документов

| Тип | Назначение | Можно ли использовать для решений | Периодичность обновления |
|---|---|---|---|
| Регламент | задает правила работы | да, как процессный источник | по решению (не чаще раза в цикл) |
| Сырые данные | фиксируют исходные данные | нет без проверки качества | пополняется при каждом сборе |
| Baseline | фиксирует состояние до изменений | да, для сравнения после внедрения | разовый (снимок на момент старта) |
| Первичный анализ | разбирает один источник | ограниченно, только в рамках источника | обновляется при новых данных |
| Матрица доказательств | объединяет факты и статусы | да | постоянно в ходе работы |
| Карта потерь/возможностей | переводит факты в бизнес-смысл | да, после матрицы | обновляется после матрицы |
| Коммерческая модель | связывает спрос, сайт, каналы, бюджет, приоритеты и результат | да, после матрицы и карты потерь | по сезонам и значимым изменениям |
| Прогноз | показывает ожидаемый диапазон результата по сценариям | да, если указаны период, сегмент, условия и уровень уверенности | по сценариям и новым данным |
| Карта решений | фиксирует управленческий выбор | да, если есть основание | пополняется (новые решения — новые записи) |
| ТЗ | передает задачи исполнителю | да, если есть критерии приемки | разовый на задачу |
| Отчет бизнесу | управленческое резюме | да, если основан на матрице и модели | регулярный (ежемесячно / ежеквартально) |
| Журнал изменений | фиксирует внедрения | да, для контроля | пополняется постоянно |
| Карта влияния | оценивает эффект изменений | да, с учетом уверенности и ограничений | обновляется после измерения результата |

> **Примечание:** В frontmatter всех аналитических инструментов стоит `type: регламент`. Этот индекс классифицирует документы по роли и назначению — это не то же самое, что значение `type` в frontmatter. Типы выше — это функциональные роли в рамках фреймворка.
>
> **Правило применимости для решений:** Столбец "Можно ли использовать для решений" действует только при условии `evidence_status: checked` или `final`. Документы со статусом `evidence_status: draft` не являются основанием для управленческих выводов. Критерии перехода между статусами — в `[[QA Gate контроль качества]]`.

## Правило: чеклист при создании документа

Документ не должен "лежать рядом". При создании нового документа проверь:

- [ ] `depends_on` заполнен — указан минимум один этап флоу или документ-источник
- [ ] `source_docs` заполнен — откуда пришли исходные данные
- [ ] `feeds` заполнен — куда уходит результат (следующий документ или этап)
- [ ] `evidence_status` выставлен корректно (`draft` → `checked` → `final`)
- [ ] Есть хотя бы одна wiki-ссылка `[[...]]` на связанный документ фреймворка
