# Выгрузка и качество данных

## Назначение

Регламент описывает, как выгружать, хранить и проверять данные по проекту, чтобы аналитика была воспроизводимой и Gate 1 флоу закрывался формальными критериями.

Главное правило: сначала сохраняется исходная выгрузка, затем проводится проверка качества данных, и только после этого делаются выводы.

**Вход:** MVA-доступы получены и верифицированы ([[Запрос доступов и данных для коллег]]).  
**Действие:** выгрузка → сохранение raw → метаинформация → проверка качества → таксономизация проблем.  
**Выход:** выгрузки со статусом `ready_for_analysis`, зафиксированные в [[Учет времени, решений и действий]].  
**Gate:** Gate 1 (Данные собраны и проверены) — см. [[QA Gate контроль качества]].

## Read-only правило

Все правила работы с доступами — в [[Запрос доступов и данных для коллег]], раздел «Правила работы с доступами». Здесь фиксируется только специфика выгрузки:

- Если метод технически `POST`, но нужен только для выгрузки (например, `POST` в Logs API Метрики для создания log-request, `POST` в Direct Reports API) — это read-only-совместимое действие. Перед запуском нужно отдельно зафиксировать в `.meta.json` выгрузки: `read_only: true, reason: "POST-метод используется для выгрузки данных, не меняет настройки проекта"`.

## Где хранить данные

Структура папок проекта:

```text
<project>/_data/
├── metrika/{YYYY-MM-DD}/*.json + *.meta.json
├── webmaster/{YYYY-MM-DD}/*.json + *.meta.json
├── direct/{YYYY-MM-DD}/*.json + *.meta.json
├── crm/{YYYY-MM-DD}/*.csv + *.meta.json
└── INDEX.md
```

- `_data/` — все выгрузки, разложены по источнику и дате сбора.
- `_reports/` — внешние PDF/DOCX/XLSX от агентства и клиента.
- `_baseline/` — контрольные снимки состояния до внедрения изменений.
- `_reglaments/` — регламенты фреймворка.

## Именование файлов

Единый шаблон (постфиксы авторства не используются):

```text
<source>_<account_or_counter_id>_<report>_<period_from>_<period_to>.json
```

Примеры:

```text
metrika_87177110_traffic-sources_2026-03-01_2026-03-31.json
webmaster_azt-teplica-ru_popular-queries_2026-03-01_2026-03-31.json
direct_12345678_reports-clicks_2026-03-01_2026-03-31.json
```

При коллизии (повторная выгрузка того же отчёта за тот же период) — добавляется суффикс `_v2`, `_v3` и т.д.

## Метаинформация выгрузки

Рядом с каждой выгрузкой лежит sidecar `<name>.meta.json` со следующими полями:

```json
{
  "source": "metrika | webmaster | direct | crm",
  "account_or_counter_id": "87177110",
  "host_or_host_id": "azt-teplica.ru",
  "report": "traffic-sources",
  "endpoint": "https://api-metrika.yandex.net/stat/v1/data",
  "request_params": { "ids": "...", "metrics": "...", "dimensions": "...", "date1": "...", "date2": "..." },
  "method": "GET",
  "read_only": true,
  "retrieved_at": "2026-04-17T12:34:56+03:00",
  "tz": "Europe/Moscow",
  "currency": "RUB",
  "api_version": "v1",
  "request_id": "x-request-id from response",
  "sample_share": 1.0,
  "sampled": false,
  "data_lag_hours": 12,
  "attribution": "lastsign",
  "errors": [],
  "sha256": "computed on raw file bytes",
  "status": "ready_for_analysis | warning | blocker",
  "status_detail": "sample_share=0.42 — blocker по порогу 0.5"
}
```

## Проверка качества данных

Проверка выполняется перед присвоением статуса. Пункты разделены на **blocker** (останавливают анализ), **warning** (анализ возможен с оговоркой), **info** (фиксируется в мете).

**Blocker (данные непригодны):**

- [ ] Счётчик / host_id не соответствует проекту.
- [ ] Период не совпадает с запрошенным.
- [ ] `sample_share < 0.5` при `sampled: true`.
- [ ] Ключевые отчёты пусты (0 строк при ожидаемом объёме).
- [ ] `data_lag_hours > 168` (7 дней) для периода, включающего вчерашний день.
- [ ] Ошибки API 5xx без успешного ретрая.
- [ ] Смешение тестовых, служебных и боевых данных.

**Warning (анализ возможен с явной оговоркой в мете и в [[Матрица доказательств]]):**

- [ ] `sample_share` ∈ [0.5; 0.9) при `sampled: true`.
- [ ] `data_lag_hours > 72` для свежего периода.
- [ ] Не все goal-метрики попали в выгрузку (chunking частично провалился).
- [ ] Timezone не совпадает с каноническим (`Europe/Moscow`) — требуется нормализация.
- [ ] Валюта не RUB (для денежных показателей) — требуется конвертация.

**Info (фиксируется в мете без изменения статуса):**

- [ ] `contains_sensitive_data: true` — персональные данные (clientID, IP): обезличить перед сохранением в vault.
- [ ] Ограничения `disclosure` (скрытые строки по требованиям Яндекса).
- [ ] Применяются ли Git LFS-правила: файл > 100 МБ — через LFS, > 10 МБ и не JSON — сжать (`.gz`).

Определения `sampling`, `sample_share`, `data_lag`, `attribution`, `chunk`, `disclosure`, `goal-метрика` — в [[Словарь метрик и сущностей#Технические сущности]].

## Таксономия статусов выгрузки

Каждая выгрузка имеет один статус:

| Статус | Когда присваивается | Что можно делать |
|---|---|---|
| `ready_for_analysis` | все blocker-пункты пройдены, warning зафиксированы в мете | использовать как источник в [[Матрица доказательств]] (с пометкой warning при наличии) |
| `warning:<тип>` | blocker пройдены, есть warning-отметки | использовать с оговоркой; выводы помечаются «уверенность: средняя» |
| `blocker:<тип>` | есть хотя бы один blocker | не использовать; перевыгружать или документировать в `errors[]` и переводить в деградированный режим (см. ниже) |

## Связь с MVA и режим деградации

Выгрузки по MVA-пунктам (из [[Запрос доступов и данных для коллег]]):

| MVA-пункт | Обязательные отчёты |
|---|---|
| Яндекс Вебмастер | диагностика, индексирование, popular-queries, query-analytics, broken-links |
| Яндекс Метрика + Вебвизор | traffic-sources, goals, landing-pages, search-phrases, devices, UTM |
| Яндекс Бизнес (счётчики в Метрике) | «Целевые события на Яндексе», показы/переходы/звонки/маршруты |
| Яндекс Директ | reports-clicks, keywords-performance, campaigns-summary |
| CRM | выгрузка заявок за 12+ месяцев по полям из [[Запрос доступов и данных для коллег]] |

Если MVA-доступ не получен (деградация, см. [[Запрос доступов и данных для коллег#Деградация MVA]]) — выгрузки по деградированному источнику получают `evidence_status: draft` в мете без возможности перехода в `checked` до получения доступа или подтверждения данных независимым источником.

## Definition of Done — когда данные готовы к анализу

Выгрузка переходит в статус `ready_for_analysis`, когда одновременно:

- [ ] Файл сохранён в `_data/<source>/<YYYY-MM-DD>/`.
- [ ] `.meta.json` sidecar заполнен всеми обязательными полями.
- [ ] Пройдены все blocker-проверки.
- [ ] Warning-пункты зафиксированы в `status_detail`.
- [ ] SHA-256 файла вычислен и записан в `.meta.json`.
- [ ] Запись добавлена в `_data/INDEX.md`.

Пакет выгрузок для Gate 1 закрыт, когда все MVA-отчёты имеют статус `ready_for_analysis` или `warning`, и статус всех выгрузок зафиксирован в [[Учет времени, решений и действий]].

## Статус-трекинг выгрузок

Реестр выгрузок ведётся в `_data/INDEX.md` — таблица:

| # | Источник | Отчёт | Период | Дата выгрузки | Путь | Статус | Комментарий |
|---|---|---|---|---|---|---|---|

Заполняется автоматически при сохранении новой выгрузки. Дубли одного отчёта за тот же период отмечаются последовательными версиями (`v2`, `v3` и т.д.).

## Retention и версионирование

- **Raw выгрузки (`_data/`):** хранятся постоянно. При размере > 10 МБ — сжатие `.gz` / `.tar.gz`. При размере > 100 МБ — Git LFS (`.gitattributes` в `_data/` фиксирует паттерн).
- **Baseline (`_baseline/`):** хранится постоянно, не архивируется.
- **Отчёты аналитика (`_reports/`):** хранятся 12 месяцев, затем перемещаются в `40 Archive/sessions/`.
- **Черновики и промежуточные файлы:** удаляются при завершении цикла.
- **Метаданные счётчиков:** экспорт целей, фильтров, пользователей, операций над URL — выгружается при старте проекта и при изменениях, хранится в `_baseline/`.

## Когда нужны дополнительные запросы

Дополнительная выгрузка инициируется, если:

- текущий срез не связывает проблему с бизнес-влиянием;
- нужно проверить конкретные URL из Вебмастера в Метрике (см. раздел «Приложение: справка по API Метрики»);
- нужно разделить регионы, каналы (сайт vs Яндекс Бизнес), устройства;
- нужно проверить цели по посадочным;
- нужно проверить платный трафик по UTM;
- нужно подтвердить или опровергнуть гипотезу из отчёта агентства.

Каждая дополнительная выгрузка проходит тот же чеклист качества и статусирование.

## Запрет на выводы из непроверенных данных

Выгрузки со статусом `blocker` не используются как основание для задач, рекомендаций бизнесу или публикации в адресных документах.

Выгрузки со статусом `warning` используются только с явной пометкой в [[Матрица доказательств]]: источник не прошёл полную проверку, вывод имеет уровень уверенности «средняя» или ниже.

P0-задача (см. [[Словарь метрик и сущностей#Приоритеты]]) не формируется на основании одной выгрузки со статусом ниже `ready_for_analysis` без независимого подтверждения (второй источник, повторная выгрузка через 24–72 часа, или подтверждение через другой API).

## Приложение: справка по API Метрики и других источников

> Справочный блок. Перед использованием примеров — проверить актуальность по документации Яндекса (`yandex.ru/dev/metrika`, `yandex.ru/dev/webmaster`, `yandex.ru/dev/direct`). Если API поменялось — обновить этот раздел и зафиксировать изменение в `_decisions/` как CHG-запись.

### Агрегированные и сырые данные

Большинство отчётов Метрики через API — агрегированные (сгруппированы по измерениям и метрикам).

Примеры агрегированных срезов:

- источник → визиты;
- посадочная → цели;
- поисковая фраза → посадочная → город;
- Яндекс Бизнес: поверхность → действие → день.

Сырые данные уровня визитов и хитов — через Logs API. Особенности Logs API:

- Создание log-request — через `POST /management/v1/counter/{id}/logrequests`, это создаёт задачу (не мгновенная выгрузка).
- Обработка занимает часы; при обнаружении проблемы, требующей сырых данных, создавать log-request параллельно с агрегированными запросами.
- Скачивание готового part — через `GET .../download?part=N`.
- Logs API не поддерживает сегменты и цели — только сырые поля; фильтрация по цели — постфактум.

### Измерения Метрики: `ym:s:*` vs `ym:pv:*`

В API Метрики два основных пространства измерений:

- `ym:s:*` — сессионные данные (визиты, пользователи, источники, посадочные, регионы, устройства, цели).
- `ym:pv:*` — хитовые данные (просмотры страниц, URL).

Нельзя строить основной отчёт так, будто `ym:s:startURL` и `ym:pv:URL` взаимозаменяемы:

- `ym:s:startURL` — страница входа в визит;
- `ym:pv:URL` — просмотренный URL, который мог быть не первым в визите.

Правила:

- Для входов: использовать `ym:s:startURL`.
- Для просмотров: использовать `ym:pv:URL`.
- Для сессионной связки с целями: использовать `ym:s:*` отчёт с фильтром по `ym:pv:URL`.

Дополнительные пространства: `ym:ad:*` (связка с Директом), `ym:sp:*` (поисковые фразы), `ym:ev:*` (e-commerce), `ym:up:*` (оффлайн-конверсии).

### Пример запроса Метрика Stat API

```text
GET https://api-metrika.yandex.net/stat/v1/data
?ids=<counter_id>
&metrics=ym:s:visits,ym:s:users
&dimensions=ym:s:lastsignSourceEngine
&date1=7daysAgo&date2=yesterday
&accuracy=full
&lang=ru
```

### Пример запроса Вебмастер API

```text
GET /user/{user_id}/hosts/{host_id}/search-queries/popular
```

Для query-analytics: `POST /user/{user_id}/hosts/{host_id}/query-analytics/list` с телом-фильтром.

Особенности: `popular-queries` отдаёт максимум 500 запросов на страницу; `indicator=TOTAL_SHOWS/TOTAL_CLICKS/AVG_SHOW_POSITION` — разные индикаторы, запрашиваются отдельно.

### Пример запроса Яндекс Директ API

```text
POST https://api.direct.yandex.com/json/v5/reports
```

В теле — имя отчёта, даты, поля, группировки. Поля `Cost/CostPerClick` — в валюте аккаунта; флаг `IncludeVAT` определяет учёт НДС.

### Чанкинг и лимиты

- Метрика: 20 метрик на запрос, 500 goal-метрик на запрос, 5000 строк на запрос (limit/offset, `offset <= 100000`), 100 rps на токен.
- Вебмастер: 30 rpm.
- Директ: баллы по формуле.

Правила чанкинга:

- Разбивать goal-метрики по 200 ID на чанк.
- Между чанками `sleep 0.2s`; при 429 — exponential backoff (base 2, max 5 попыток).
- Фиксировать статус каждого чанка в `manifest.json` в папке выгрузки: id чанка, параметры, статус (`ok/failed/retry`), хеш параметров.
- Если часть чанков провалилась — статус всей выгрузки `warning:incomplete_chunks` или `blocker:incomplete_chunks` в зависимости от доли провалов (< 10% — warning, ≥ 10% — blocker).

### Ошибки API и реакция

| Код | Причина | Действие |
|---|---|---|
| 429 | rate limit | exponential backoff |
| 403 | wrong scope | проверить OAuth-токен и scope |
| 400 `wrong_parameter_value` | некорректный параметр | залогировать в `errors[]`, не повторять, эскалировать |
| 500 / 503 | сервер Яндекса | повторить до 3 раз с паузой, затем залогировать как blocker |

### Нормализация URL при join Метрика ↔ Вебмастер

Перед сопоставлением `ym:s:startURL` и URL из Вебмастера выполняется нормализация:

- удалить trailing slash (или добавить — главное единообразно);
- нормализовать схему (`http://` → `https://` или оставить обе с пометкой);
- убрать query string для чистого pathname-матчинга;
- привести к одному регистру домена (`www.` / без `www.`).

Без нормализации join даёт 0 совпадений.

### Сверка Метрика ↔ Вебмастер

Классическое расхождение: клики в Вебмастере ≠ визиты из поиска в Метрике. Причины: разные методологии (impression-based vs session-based), сэмплирование, фильтры роботов.

Допустимое расхождение: 10–30% — не аномалия. Расхождение > 30% — фиксируется как `warning:source_mismatch` в [[Матрица доказательств]] с указанием гипотезы причины.
