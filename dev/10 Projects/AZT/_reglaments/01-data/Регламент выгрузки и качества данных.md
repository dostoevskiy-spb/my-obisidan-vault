# AZT — Регламент выгрузки и качества данных

## Назначение

Регламент описывает, как выгружать, хранить и проверять данные по проекту AZT, чтобы аналитика была воспроизводимой.

Главное правило: сначала сохраняется исходная выгрузка, затем проводится проверка качества данных, и только после этого делаются выводы.

## Read-only правило

Для Яндекс Вебмастера, Яндекс Метрики и других аналитических систем по умолчанию разрешены только read-only действия.

Запрещено без отдельного прямого согласия:

- отправлять URL на переобход;
- добавлять или удалять sitemap;
- менять региональность;
- менять настройки счетчиков;
- менять цели;
- менять доступы;
- запускать рекламные изменения;
- удалять счетчики или связи.

Если метод технически использует `POST`, но нужен только для выгрузки данных, например Logs API, перед запуском нужно отдельно зафиксировать, что метод не меняет настройки проекта.

## Где хранить данные

Все выгрузки кладутся в:

```text
10 Projects/AZT/_data
```

Внешние PDF, сканы и экспортированные отчеты кладутся в:

```text
10 Projects/AZT/_reports
```

Контрольные снимки состояния до исправлений кладутся в:

```text
10 Projects/AZT/_baseline
```

Регламенты кладутся в:

```text
10 Projects/AZT/_reglaments
```

## Именование

Все новые файлы Codex по проекту AZT должны иметь постфикс `-cx` перед расширением.

Пример:

```text
AZT — Выгрузка Яндекс Метрики 2026-04-16-cx.json
AZT — Согласование Метрики и Вебмастера 2026-04-16-cx.md
```

## Обязательная метаинформация выгрузки

В каждой выгрузке или сопроводительном описании должны быть:

- дата выгрузки;
- период данных;
- источник;
- ID счетчика, сайта или хоста;
- метод получения;
- список отчетов внутри файла;
- ограничения данных;
- ошибки API, если были;
- указание, были ли данные агрегированными или сырыми.

## Проверка качества данных

Перед анализом нужно проверить:

- правильный ли счетчик;
- правильный ли период;
- не перепутаны ли счетчик сайта и счетчики Яндекс Бизнеса;
- есть ли ошибки API;
- не пустые ли ключевые отчеты;
- есть ли sampling;
- `sampled`;
- `sample_share`;
- `sample_size`;
- `sample_space`;
- `total_rows`;
- `total_rows_rounded`;
- `contains_sensitive_data`;
- `data_lag`;
- `attribution`;
- есть ли ограничения disclosure;
- совпадает ли timezone;
- совпадает ли валюта, если анализируются денежные показатели;
- есть ли цели и какие из них рабочие;
- нет ли смешения тестовых, служебных и боевых данных;
- применяются ли Git LFS правила для больших JSON/PDF.

## Агрегированные и сырые данные

Большинство отчетов Метрики через API являются агрегированными: они сгруппированы по измерениям и метрикам.

Примеры агрегированных срезов:

- источник → визиты;
- посадочная → цели;
- поисковая фраза → посадочная → город;
- Яндекс Бизнес: поверхность → действие → день.

Сырые данные уровня визитов и хитов требуют отдельного Logs API. Использовать его нужно только после того, как агрегированные данные перестали отвечать на вопрос.

## Совместимость измерений и метрик Метрики

В API Метрики есть разные наборы данных.

`ym:s:*` — сессионные данные. Они используются для визитов, пользователей, источников, посадочных страниц, регионов, устройств и целей.

`ym:pv:*` — хитовые данные. Они используются для просмотров страниц и конкретных просмотренных URL.

Нельзя строить основной отчет так, будто `ym:s:startURL` и `ym:pv:URL` взаимозаменяемы:

- `ym:s:startURL` — страница входа в визит;
- `ym:pv:URL` — просмотренный URL, который мог быть не первым в визите.

Если нужно проверить проблемный URL из Вебмастера:

- для входов использовать `ym:s:startURL`;
- для просмотров использовать `ym:pv:URL`;
- для сессионной связки с целями использовать `ym:s:*` отчет с фильтром по `ym:pv:URL`.

## Лимиты и чанкинг

Если в запросе много goal-метрик, их нужно разбивать на чанки.

Правило:

- не считать выгрузку целей полной, если часть goal-метрик не попала в запрос;
- сохранять список goal ID, которые вошли в каждый chunk;
- фиксировать ошибки API по каждому chunk;
- после выгрузки проверять, что все важные цели покрыты.

## Когда нужны дополнительные запросы

Дополнительные запросы нужны, если:

- текущий срез не связывает проблему с бизнес-влиянием;
- нужно проверить конкретные URL из Вебмастера;
- нужно разделить регионы;
- нужно разделить сайт и Яндекс Бизнес;
- нужно проверить цели по посадочным;
- нужно проверить платный трафик по UTM;
- нужно проверить мобильный трафик отдельно;
- нужно подтвердить или опровергнуть гипотезу из отчета агентства.

## Запрет на выводы из непроверенных данных

Если выгрузка неполная, содержит ошибки API или не покрывает нужный период, выводы по ней маркируются как `требует проверки`.

Нельзя использовать неполную выгрузку как основание для P0-задачи, если нет другого независимого подтверждения.
