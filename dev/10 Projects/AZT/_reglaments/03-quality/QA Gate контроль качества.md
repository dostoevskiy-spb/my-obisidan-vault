# QA Gate контроль качества

## Назначение

QA Gate нужен, чтобы слабые данные, непроверенные выводы, ошибки агентства, неточные формулировки и неподтвержденные управленческие рекомендации не проходили дальше по флоу.

Принцип:

```text
нет проверки gate → нет перехода на следующий этап → нет финального решения
```

## Статусы gate

| Статус | Значение |
|---|---|
| `pass` | проверка пройдена, можно переходить дальше |
| `conditional pass` | можно идти дальше, но с явными ограничениями и открытыми вопросами |
| `blocked` | переход запрещен до устранения блокера |
| `fail` | данные или вывод непригодны для текущего решения |

`conditional pass` нельзя использовать для финальных управленческих выводов без блока ограничений.

## Gate 1. Данные собраны и проверены

Цель: убедиться, что исходные данные можно использовать.

Проверяется:

- источник данных известен;
- период указан;
- права доступа понятны;
- счетчик, хост, кампания или документ идентифицированы;
- API не вернул критические ошибки;
- выгрузка не перепутана между сайтом, Яндекс Бизнесом, Директом и CRM;
- сырые данные сохранены;
- ограничения источника зафиксированы;
- RACI заполнен хотя бы в части владельцев данных;
- есть список недостающих данных.

Типовые блокеры:

- неизвестно, какой счетчик используется;
- период не совпадает между источниками;
- API вернул ошибку или неполные данные;
- нет доступа к критичному источнику;
- данные агентства представлены без первичных выгрузок и доступа к кабинету.

Выход gate:

- сырые выгрузки;
- baseline;
- список ограничений;
- список недостающих данных;
- черновой RACI;
- статус `pass`, `conditional pass` или `blocked`.

## Gate 2. Первичные анализы готовы

Цель: убедиться, что каждый источник разобран отдельно и не подменяет другой источник.

Проверяется:

- по каждому источнику есть отдельный вывод;
- выводы не выходят за пределы источника;
- лиды, микроконверсии и верх воронки не смешаны;
- абсолютные значения указаны вместе с процентами;
- период и фильтры указаны;
- странности и аномалии вынесены в вопросы или ограничения;
- отчеты агентства не приняты как факт без сверки с доступами.

Типовые блокеры:

- вывод по Метрике делается на основании Яндекс Бизнеса без оговорки;
- отчет агентства противоречит данным кабинета;
- проценты используются без базы;
- неясно, что означает цель или событие.

Выход gate:

- первичные анализы источников;
- список конфликтов;
- список гипотез;
- список проверок для матрицы доказательств.

## Gate 3. Матрица доказательств согласована

Цель: отделить доказанные факты от гипотез и подготовить основу для приоритетов.

Проверяется:

- каждый сильный вывод есть в матрице доказательств;
- у каждого вывода есть источник и период;
- статус доказанности указан;
- альтернативные объяснения рассмотрены;
- противоречия между источниками вынесены отдельно;
- технические, SEO, рекламные и бизнес-выводы не смешаны;
- словарь метрик используется последовательно;
- предварительные P0/P1/P2 не противоречат фактам.

Типовые блокеры:

- вывод не имеет строки в матрице;
- факт и гипотеза написаны одинаковым тоном;
- есть конфликт источников без объяснения;
- приоритет назначен без доказанного основания.

Выход gate:

- матрица доказательств;
- карта потерь и возможностей;
- список открытых вопросов;
- предварительные приоритеты;
- черновая модель приоритизации.

## Gate 4. Коммерческая модель и адресные документы готовы

Цель: убедиться, что выводы можно отдавать собственнику, агентству или исполнителям.

Проверяется:

- коммерческая модель опирается на доказанные данные;
- приоритеты рассчитаны или объяснены через модель приоритизации;
- у задач есть владельцы по RACI;
- у задач есть критерии приемки;
- отчет собственнику не содержит технического шума без бизнес-смысла;
- ТЗ разработчику не содержит управленческой риторики;
- вопросы агентству сформулированы по фактам;
- SEO-статьи не запускаются без спроса, роли в воронке и технической готовности;
- рекомендации по Директу не даются без посадочных, целей, UTM и способа проверки.

Типовые блокеры:

- нет владельца `A` или исполнителя `R`;
- приоритеты не объяснены;
- в отчете есть неподтвержденные обвинения;
- в ТЗ нет критерия приемки;
- в рекомендациях по деньгам нет данных о деньгах или явного ограничения.

Выход gate:

- финальные или условно финальные P0/P1/P2;
- отчет собственнику;
- ТЗ;
- вопросы агентству;
- план ближайшего цикла.

## Gate 5. Внедрение проверено и влияние оценено

Цель: закрыть цикл и понять, сработали ли решения.

Проверяется:

- изменение зафиксировано в журнале изменений;
- есть baseline до изменения;
- дата проверки указана;
- результат проверен источником, а не словами "сделано";
- карта влияния заполнена после контрольного окна;
- учтены внешние факторы: сезонность, бюджет, параллельные изменения, индексация, технические сбои;
- матрица доказательств и приоритеты обновлены;
- прогресс флоу обновлен.

Типовые блокеры:

- исполнитель сказал "сделано", но проверки нет;
- нет baseline;
- прошло недостаточно времени для оценки влияния;
- одновременно было несколько изменений и эффект нельзя разделить;
- ухудшение показателя не расследовано.

Выход gate:

- отчет контроля после внедрения;
- обновленная карта влияния;
- обновленная матрица доказательств;
- обновленные приоритеты;
- решение о следующем цикле.

## Шаблон QA Gate

```markdown
## QA Gate X — название

| Проверка | Статус | Источник | Кто проверил | Комментарий |
|---|---|---|---|---|
|  | pass / conditional pass / blocked / fail |  |  |  |

Итоговый статус gate: `...`

Блокеры:

- ...

Ограничения:

- ...

Решение:

- переход разрешен / переход запрещен / переход разрешен с ограничениями.
```

## Главное правило

Если gate не пройден, документ остается рабочим. Его нельзя использовать как финальную основу для бюджета, задач, обвинений, приоритетов или обещаний результата.
