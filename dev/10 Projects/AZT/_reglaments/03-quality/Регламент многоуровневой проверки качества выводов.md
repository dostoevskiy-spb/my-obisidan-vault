# AZT — Регламент многоуровневой проверки качества выводов

## Назначение

Этот регламент задает обязательные проверки на каждом этапе работы с данными, документами, выводами и задачами по проекту AZT.

Цель: не допускать в документы неточности, неподтвержденные утверждения, неверные выводы, технические ошибки в расчетах и некорректные управленческие рекомендации.

## Главный принцип

Каждый этап должен завершаться проверкой.

Нельзя переходить к следующему этапу, если на текущем:

- неизвестен источник данных;
- не проверен период;
- есть ошибки API;
- неясно, что именно означает метрика;
- вывод не связан с доказательством;
- цифра не воспроизводится;
- формулировка может быть понята неверно;
- нет критерия проверки после исправления.

## Уровни проверки

### 1. Проверка источника

Перед использованием данных нужно проверить:

- источник;
- дату выгрузки;
- период;
- ID счетчика, хоста, кампании или документа;
- права доступа;
- тип данных: сырые, агрегированные, отчет, ручная проверка;
- применимость источника к задаче.

Пример: счетчик сайта и счетчик Яндекс Бизнеса нельзя использовать как один источник, потому что они отвечают на разные вопросы.

### 2. Проверка API и выгрузки

Для API-данных проверяется:

- успешность запроса;
- отсутствие ошибок API;
- `sampled`;
- `sample_share`;
- `total_rows`;
- `total_rows_rounded`;
- `contains_sensitive_data`;
- `data_lag`;
- `timezone`;
- `attribution`;
- полнота выгрузки;
- покрытие всех нужных целей;
- корректность chunk-выгрузки;
- отсутствие смешения `ym:s:*` и `ym:pv:*` вне правил.

Если есть ошибка API или неполная выгрузка, вывод получает статус `требует проверки`.

### 3. Проверка смысла метрики

Перед выводом нужно ответить:

- что именно измеряет метрика;
- что она не измеряет;
- является ли она лидом, микроконверсией, коммерческим действием или верхом воронки;
- можно ли сравнивать ее между счетчиками;
- можно ли суммировать ее с другими метриками;
- не является ли она технической или автоматической.

Пример: `Показ организации` в Яндекс Бизнесе показывает видимость карточки, но не является заявкой.

### 4. Проверка независимым источником

Сильный вывод должен быть подтвержден минимум двумя разными типами источников, если это возможно.

Примеры связок:

- Вебмастер + Метрика;
- Вебмастер + ручная проверка URL;
- Метрика + Вебвизор;
- Wordstat + Вебмастер + Метрика;
- Яндекс Бизнес + региональная посадочная + Вебмастер;
- медиаплан агентства + UTM + Метрика;
- SEO-ТЗ агентства + Wordstat + конкурентная выдача.

Если независимого источника нет, вывод остается гипотезой.

### 5. Проверка альтернативных объяснений

Перед финальным выводом нужно проверить, нет ли другого объяснения.

Примеры:

- падение трафика может быть сезонностью, а не только технической ошибкой;
- нулевая цель может быть не отсутствием заявок, а неправильной настройкой цели;
- отсутствие SEO-трафика может быть связано с новым разделом, а не плохим контентом;
- рост показов карточки может быть результатом рекламы, а не органического спроса;
- плохая конверсия мобильного трафика может быть следствием качества трафика, а не только UX.

Если альтернативное объяснение возможно, оно фиксируется в открытых вопросах.

### 6. Проверка расчета

Все расчеты должны быть воспроизводимыми.

Для каждого расчета фиксируется:

- формула;
- период;
- источник;
- фильтры;
- сегменты;
- округление;
- исключения.

Нельзя использовать проценты без абсолютных значений.

Корректно:

```text
34 перехода на сайт из карточки за 09.04.2026–15.04.2026.
```

Некорректно:

```text
Переходы просели на 19%, без абсолютной базы и периода.
```

### 7. Проверка формулировки

Каждый сильный вывод должен быть написан так, чтобы его нельзя было понять шире, чем позволяют данные.

Запрещены формулировки:

- "доказано", если есть только один источник;
- "теряем деньги", если нет данных о деньгах;
- "не работает", если есть только частичная проблема;
- "пользователи не хотят", если есть только поведенческий сигнал;
- "агентство ошиблось", если есть только несогласованность без проверки.

Разрешенные формулировки:

- "данные показывают";
- "по текущей выгрузке";
- "в проверенной выборке";
- "есть риск";
- "гипотеза требует проверки";
- "вывод подтвержден таким-то источником".

### 8. Проверка адресата

Один и тот же факт формулируется по-разному:

- собственнику — через риск, деньги, порядок действий;
- разработчику — через URL, правило, критерий приемки;
- агентству — через вопрос, KPI, источник;
- аналитику — через метод, фильтр, период и статус.

Перед отправкой нужно проверить, что документ соответствует адресату.

### 9. Проверка действия

Каждый вывод, который превращается в задачу, должен иметь:

- конкретное действие;
- владельца или тип исполнителя;
- приоритет;
- критерий готовности;
- способ проверки;
- контрольный срок или период перепроверки.

Если действие невозможно сформулировать, вывод не готов для финального документа.

### 10. Проверка после исправления

После внедрения исправления нужно проверить:

- технический результат;
- Вебмастер после переобхода;
- Метрику после контрольного периода;
- цели;
- проблемные URL;
- региональные страницы;
- карточки Яндекс Бизнеса, если затронут регион;
- рекламные посадочные, если затронут платный трафик.

Исправление считается закрытым только после проверки результата, а не после сообщения "сделано".

## Stop-критерии

Работу нужно остановить и не включать вывод в финальный документ, если:

- данные противоречат друг другу и причина не найдена;
- источник неясен;
- API вернул ошибку;
- период не совпадает;
- счетчик не тот;
- цель не классифицирована;
- вывод требует данных, которых нет;
- формулировка может привести к неверному управленческому решению.

## Минимальный чеклист перед публикацией

Перед публикацией любого важного документа проверить:

- все цифры имеют источник и период;
- все сильные выводы есть в матрице доказательств;
- все гипотезы отделены от фактов;
- нет эмоциональных оценок;
- нет неподтвержденных обвинений;
- нет смешения сайта, Яндекс Бизнеса и рекламы;
- нет смешения лидов, микроконверсий и верхнего уровня воронки;
- есть список открытых вопросов;
- есть критерии проверки после действий.

## Итоговое правило

Если вывод нельзя защитить перед собственником, агентством или разработчиком ссылкой на источник, цифру и метод проверки, он не должен попадать в финальный документ.
