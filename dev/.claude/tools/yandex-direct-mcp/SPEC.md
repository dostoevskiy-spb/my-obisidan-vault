# Спецификация приложения Direct MCP

## 1. Общее описание

**Название:** Direct MCP
**Тип:** Локальный инструмент (MCP-сервер) для ассистента Claude Code.
**Разработчик:** Ламзин Павел (`pavel-lamzin`, `pavel.lamzin@yandex.com`).
**Назначение:** Аудит и анализ рекламных кампаний в Яндекс.Директе через ассистент Claude Code, без изменения данных в кабинете.

## 2. Задачи, которые решает приложение

- Сверка структуры кампаний, групп, объявлений и ключевых фраз с согласованным медиапланом.
- Контроль соответствия фактических ставок, корректировок и таргетингов утверждённой стратегии.
- Получение статистики по кампаниям, объявлениям и поисковым запросам для еженедельной отчётности.
- Отслеживание изменений, внесённых агентством (через `Changes.check*`).
- Инвентаризация креативов, расширений, минус-слов и аудиторий.

## 3. Архитектура

- **Клиент:** локальный процесс Python 3.12 на рабочей машине разработчика. Публичного сервиса нет, приложение не работает в автоматическом режиме, все запросы инициируются пользователем вручную.
- **Транспорт:** HTTPS POST к `https://api.direct.yandex.com/json/v5/{service}` (production) или `https://api-sandbox.direct.yandex.com/json/v5/{service}` (песочница).
- **Авторизация:** OAuth 2.0 Bearer-токен, хранится локально в файле `.credentials.md`, исключённом из системы контроля версий. Токен в код приложения не попадает.
- **Учёт данных:** ответы API кешируются локально (TTL 60 секунд для повторных get, длинные ответы складываются в файловый кеш пользователя). Данные не передаются третьим лицам и не покидают локальную машину.

Приложение использует **только методы чтения**. Методы, изменяющие данные (`add`, `update`, `delete`, `set`, `suspend`, `resume`, `archive`, `unarchive`, `moderate`), физически отсутствуют в кодовой базе.

## 5. Схема и последовательность вызовов

### Сценарий A. Плановый аудит кампаний (1 раз в неделю)

1. `Clients.get` — получение параметров аккаунта (валюта, ограничения).
2. `Campaigns.get` — полный список кампаний.
3. Для списка `CampaignIds`, полученного на шаге 2:
   1. `AdGroups.get(CampaignIds=...)`
   2. `Ads.get(CampaignIds=...)`
   3. `Keywords.get(CampaignIds=...)`
4. `BidModifiers.get(CampaignIds=...)` — корректировки ставок.
5. `AdExtensions.get`, `Sitelinks.get`, `VCards.get` — расширения и визитки.
6. `NegativeKeywordSharedSets.get`, `RetargetingLists.get` — минус-слова и аудитории.

### Сценарий B. Мониторинг изменений агентства (ежедневно по запросу пользователя)

1. `Changes.checkDictionaries(Timestamp=T)` — проверка справочников.
2. `Changes.checkCampaigns(Timestamp=T)` — список изменённых `CampaignIds`.
3. Если на шаге 2 список непустой, вызывается `Changes.check(CampaignIds=[...], Timestamp=T, FieldNames=[CampaignIds, AdGroupIds, AdIds, TargetIds])` батчами по 1000 ID. Timestamp не дальше 7 дней в прошлое.

### Сценарий C. Отчёт по статистике (1–3 раза в неделю)

1. Формирование параметров запроса: `ReportType`, `DateFrom`, `DateTo`, `FieldNames`, `IncludeVAT`, `IncludeDiscount`.
2. POST на `Reports.getReport` с заголовками `processingMode: auto`, `returnMoneyInMicros: false`, `skipReportHeader: true`, `skipColumnHeader: false`, `skipReportSummary: true`.
3. При получении статуса 201 или 202 — ожидание значения, указанного в заголовке `retryIn` ответа, и повторный запрос с тем же телом. Лимит времени ожидания — 600 секунд.
4. При получении статуса 200 — сохранение TSV-тела в локальный файл, парсинг первых 10 строк для предпросмотра.

### Сценарий D. Работа со справочниками (по необходимости)

`Dictionaries.get(DictionaryNames=[...])` — одиночный запрос при необходимости декодирования ID регионов, часовых поясов, валют и т. п. Результат кешируется локально на 60 секунд.

## 6. Частота вызовов

Приложение работает в интерактивном режиме, все запросы инициируются пользователем. Фоновых и циклических вызовов нет. Ниже — оценка средней нагрузки при регулярной работе с несколькими рекламными аккаунтами и ежедневной проверке текущего состояния кампаний.

|---|---|

| `AdGroups.get` | до 40 вызовов в сутки |

## 7. Обработка ошибок

Приложение разделяет ошибки на три категории с разной стратегией реакции.

### 7.1. Ошибки, подлежащие повторной попытке (retry)

**Условия для retry:**
- Коды ошибок API: `52` (Превышено число параллельных запросов), `152` (Превышен лимит запросов в секунду).
- HTTP-статусы транспортного уровня: `500`, `502`, `503`, `504`.

**Алгоритм retry:**
- Максимум 3 попытки подряд.
- Задержка между попытками: значение из заголовка `Retry-After` ответа, если оно есть; иначе экспоненциальный backoff 1, 2, 4 секунды.
- Если все 3 попытки исчерпаны, вызов прерывается с ошибкой, пользователю возвращается человекочитаемое сообщение.

### 7.2. Ошибки авторизации и валидации — без retry

Ошибки с кодами `1002` (invalid auth), `58` (invalid OAuth), `4001` (invalid SelectionCriteria), `53` (no Client-Login) и прочие ошибки конфигурации возвращаются пользователю немедленно. Retry в таких случаях бесполезен и нежелателен. Сообщение об ошибке включает код, текст из `error_string`, `error_detail` и, при коде 53, подсказку про отсутствующий заголовок `Client-Login`.

### 7.3. Reports API — polling

При статусе ответа 201 или 202 приложение ожидает значение из заголовка `retryIn` (или `Retry-After`, либо 5 секунд по умолчанию) и повторяет запрос. Повторы продолжаются до статуса 200, максимум 600 секунд суммарно, после чего вызов прерывается.

### 7.4. Логирование

Все исходящие вызовы и их результаты (включая ошибки и списанные units) записываются локально в append-only файл `cache/audit.log` в формате JSON Lines. Это позволяет в любой момент восстановить историю обращений и объём потраченных units.

## 8. Учёт ограничений API Директа

### 8.1. Квота units

- Приложение парсит поле `units` из ответа каждого запроса и поддерживает in-process счётчик: `rest`, `daily_limit`, `spent` за сессию.
- Перед каждым запросом проверяется остаток: если `rest < 1000`, выполнение запроса прекращается с ошибкой `«Низкий остаток units — останавливаюсь»`. Это исключает полное израсходование суточной квоты.
- Публикуется отдельная команда `direct_units_status`, которая одним дешёвым запросом (`Campaigns.get` с `Page.Limit=1`) возвращает текущий остаток units и даёт пользователю возможность проверить квоту до начала крупных операций.
- Оценка суточного расхода с учётом тарифов API Директа («фиксированная стоимость метода + стоимость за объекты», см. документацию):
  - Плановый аудит кабинета (сценарий A): ~1 000–2 000 units за запуск.
  - Мониторинг изменений (сценарий B): ~50–150 units за цикл.
  - Построение отчётов (сценарий C): отдельная квота отчётов, общие units расходуются незначительно.
  - Работа со справочниками (сценарий D): до 100 units за вызов.
- Ожидаемый суточный расход — **в пределах 15 000–20 000 units**. Фактический расход меньше за счёт локального кеша повторных get-вызовов (§8.5) и ограничения пагинации по умолчанию (§8.2).

### 8.2. Постраничный вывод

- `Page.Limit` = 1000 элементов.
- Hard-cap количества страниц — 100 на один вызов (предохранитель от бесконечных циклов).
- Максимальное количество возвращаемых элементов по умолчанию — 2000 (~2 страницы). Для полного дампа пользователь должен явно передать параметр `fetch_all=True`.
- Цикл пагинации прерывается, если длина полученного чанка меньше `Page.Limit` или если остаток units приблизился к порогу.

### 8.3. Ограничение на число одновременных запросов

Приложение выполняет запросы последовательно, одним клиентом, без параллелизма — тем самым соблюдается лимит Директа (не более 5 одновременных запросов на рекламодателя), параллельных вызовов не возникает вовсе.

### 8.4. Ограничение окна `Changes` в 7 дней

В композитном сценарии мониторинга изменений (`Changes.checkCampaigns` → `Changes.check`) приложение принудительно ограничивает Timestamp семью сутками в прошлое. При запросе бо́льшего окна используется скользящее окно по 7 суток, без попытки выйти за допустимое API значение.

### 8.5. Повторные get-вызовы в рамках сессии

Для get-методов реализован in-memory кеш с TTL 60 секунд, ключ — `(service, method, хеш параметров, Client-Login)`. Повторный вызов с теми же параметрами в ближнем временном окне не обращается к API и не тратит units. Отчёты (`Reports.getReport`) из этого кеша исключены — данные всегда свежие.

### 8.6. Ограничения Reports API

- Типы отчётов, параметры `IncludeVAT`, `IncludeDiscount`, `Goals`, `AttributionModels`, `Filter`, `OrderBy` задаются при создании отчёта.
- Polling строго по значению `retryIn` из ответа API — фиксированный интервал не используется.
- Ограничение времени ожидания — 600 секунд, после чего вызов прерывается. Это исключает блокирование очереди Reports со стороны клиента.
