# vault-digest

Обработка сырых логов сессий — создание детальных заметок с архитектурой, деталями реализации, wiki-links и обновление ADR.

## Workflow

### 1. Найти необработанные файлы

Прочитай все `.md` файлы в `01 Inbox/sessions/raw/` где frontmatter содержит `status: unprocessed`.
Если таких нет — сообщи "Нет новых логов для обработки" и завершись.

### 2. Определить контекст проекта

Для каждого raw-файла:

**a) Определи субмодуль:**
- Из заголовка: `# Сессия: [api] ...` → субмодуль = `api`
- Из упоминаний путей файлов в логе: `app/Modules/Export/` → субмодуль = `api`
- Из контекста (Laravel, Filament → api; Symfony → backend; Next.js, React → front)
- Если неясно → `общий`

**b) Прочитай CLAUDE.md проекта:**
Из поля `source` в frontmatter определи директорию проекта. Прочитай:
- `{project_dir}/CLAUDE.md` — общий контекст проекта (стек, архитектура, модули)
- `{project_dir}/{submodule}/CLAUDE.md` — контекст субмодуля (если существует)

Это даёт: стек технологий, ключевые компоненты, зависимости — для более точного саммари.

### 3. Обработать каждый файл

#### Тип: session-log-raw (логи сессий)

**Извлечь:**
- Заголовок, цель, дата, проект, субмодуль
- Секция `## Итоги` — что выполнено
- Секция `## Реализация` — детали реализации (таблицы, API, компоненты)
- Секция `## Расхождения с планом` — если есть
- Записи `### HH:MM` из `## Лог` — таймлайн действий
- Проблемы и их решения

**Если нет секции `## Реализация`** (старый формат): извлеки детали из записей Лога — там часто упоминаются таблицы, файлы, эндпоинты. Структурируй их.

**Создать саммари:**

Проверка дублей: ищи в `10 Projects/{Project}/` и `01 Inbox/` заметку с тем же `source`. Если найдена — ОБНОВИ.

Имя файла: `{submodule}--{session-date}--{slug}.md`
Папка: `10 Projects/{Project}/{submodule}/` (создать если нет)
Если проекта нет в vault: `01 Inbox/`

```yaml
---
type: session-log
project: {project}
submodule: {submodule}
source: {source из raw}
session_date: YYYY-MM-DD
tags: [session-log, {project}, {submodule}]
created: YYYY-MM-DD
---
```

```markdown
# [{submodule}] {Заголовок сессии}

**Проект:** [[{Project}]]
**Субмодуль:** {submodule} ({краткое описание стека из CLAUDE.md})
**Дата:** YYYY-MM-DD

## Бизнес-контекст
{Зачем это нужно — какую проблему или задачу решает. 1-2 предложения.}

## Цель
{Из **Цель:** в логе}

## Что реализовано

### База данных
{Таблицы с полями и типами, миграции. Пропустить если нет.}

### API / Эндпоинты
{Method + path + описание. Пропустить если нет.}

### Ключевые компоненты
{Классы, сервисы, jobs с namespace и назначением. Пропустить если нет.}

### UI / Frontend
{Компоненты, страницы, изменения интерфейса. Пропустить если нет.}

### Конфигурация
{Новые настройки, миграции конфигов. Пропустить если нет.}

### Тесты
{Количество, что покрывают, ключевые assertions.}

## Архитектурные решения
{Каждое решение: что решено, почему, альтернативы.
Эта секция ОБЯЗАТЕЛЬНА если в логе есть выбор технологий, паттернов, подходов.}

## Проблемы и решения
{Если были — симптом, причина, решение. Пропустить если не было.}

## Расхождения с планом
{Если были — план vs реализация. Пропустить если не было плана.}

## Связи
- [[{Project}]]
- [[{другие связанные заметки}]]
```

#### Тип: session-plan-raw (планы)

Планы обрабатываются иначе — из них извлекается архитектура.

Имя: `{submodule}--plan--{session-date}--{slug}.md`
Папка: `10 Projects/{Project}/{submodule}/`

```yaml
---
type: session-plan
project: {project}
submodule: {submodule}
source: {source}
session_date: YYYY-MM-DD
tags: [session-plan, {project}, {submodule}]
created: YYYY-MM-DD
---
```

```markdown
# [{submodule}] План: {Название}

**Проект:** [[{Project}]]
**Субмодуль:** {submodule}
**Дата:** YYYY-MM-DD

## Архитектура
{Стек, компоненты, БД, интеграции — извлечь из плана}

## Фазы
{Пронумерованный список фаз с кратким описанием каждой}

## Ключевые решения
{Технические решения принятые в плане — что и почему}

## Ограничения и зависимости
{Если указаны в плане}

## Связи
- [[{Project}]]
```

### 4. Обновить ADR

Если в обработанной сессии или плане обнаружены архитектурные решения:

**Файл:** `10 Projects/{Project}/{submodule}/adr.md`

Если файл не существует — создать с frontmatter:
```yaml
---
type: adr
project: {project}
submodule: {submodule}
tags: [adr, {project}, {submodule}]
updated: YYYY-MM-DD
---

# ADR: {Project} — {submodule}

Архитектурные решения модуля.

## Решения
```

Добавить новые решения (проверить что такое решение ещё не записано):
```markdown
### YYYY-MM-DD: {Краткое название решения}
**Контекст:** {Почему возник вопрос}
**Решение:** {Что выбрано}
**Альтернативы:** {Что рассматривалось}
**Статус:** принято / реализовано / отменено
```

### 5. Архивировать raw

1. Обнови frontmatter raw-файла: `status: processed`
2. Перемести из `01 Inbox/sessions/raw/` в `40 Archive/sessions/`

### 6. Итоговый отчёт

Покажи:
- Сколько обработано (сессий / планов)
- Какие проекты и субмодули затронуты
- Созданные/обновлённые заметки (пути)
- Обновлены ли ADR (какие решения добавлены)
- Найденные связи

## Правила

- Саммари должно быть понятно ПОЛНОСТЬЮ без чтения исходного лога
- Перечисляй конкретику: таблицы с полями, эндпоинты с methods, классы с namespace
- Бизнес-контекст обязателен — зачем это нужно
- Архитектурные решения выделяй отдельно и дублируй в ADR
- Проблемы и решения — конкретные, с root cause
- Язык — русский
- Wiki-links: `[[Название]]`
- Не создавать дубли (проверять source в frontmatter)
- При обновлении существующей заметки — сохранять ручные правки
